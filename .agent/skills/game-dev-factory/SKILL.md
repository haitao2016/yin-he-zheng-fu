# Game Dev Factory — 多角色协作开发工厂

> 将 ChatDev 多 Agent 协作模式适配为单 Agent 角色轮转对话链，
> 7 个虚拟角色按序推导，内嵌自我修正循环，交付零构建错误的游戏代码。

---

## §1 核心概念

### 1.1 角色链（Role Chain）

7 个虚拟角色按固定顺序推导，每个角色的输出是下一角色的输入：

```
R1 制作人 → R2 策划 → R3 架构师 → R4 程序员 → R5 审查员 → R6 测试员 → R7 终审员
```

| 编号 | 角色 | 英文 | 核心职责 | 输出物 |
|------|------|------|---------|--------|
| R1 | 制作人 | Producer | 需求澄清、范围界定、风险识别 | 需求规格书 |
| R2 | 策划 | Designer | 玩法设计、数值框架、用户体验 | 策划文档 |
| R3 | 架构师 | Architect | 技术选型、模块拆分、文件结构 | 架构设计 |
| R4 | 程序员 | Programmer | 编写代码、实现功能 | 源代码文件 |
| R5 | 审查员 | Reviewer | 代码审查、规范检查、优化建议 | 审查报告 |
| R6 | 测试员 | Tester | 构建验证、运行测试、问题检测 | 测试报告 |
| R7 | 终审员 | Final Reviewer | 最终验收、交付确认 | 验收报告 |

### 1.2 自我修正循环（Self-Correction Loop）

当 R6（测试员）检测到构建失败时，触发修正循环：

```
R6 检测到错误
  ↓
分析错误原因 → 生成修正建议
  ↓
R4（程序员）根据建议修复代码
  ↓
重新调用 build 工具
  ↓
R6 再次验证
  ↓
├─ 通过 → 继续到 R7
└─ 失败 → 重复修正（最多 3 轮）
     ↓
     超过 3 轮 → 标记为 failed，报告无法自动修复的问题
```

**修正轮次上限**: 3 轮。超过后停止修正，将详细错误信息报告给用户。

### 1.3 状态追踪

项目状态通过两个文件追踪：

| 文件 | 格式 | 用途 |
|------|------|------|
| `docs/factory-status.md` | Markdown | 人类可读的进度报告 |
| `docs/factory-state.json` | JSON | 机器可读的状态数据 |

**项目状态枚举**:

```
idea → designing → developing → reviewing → testing → correcting → finalizing → completed → failed
```

## §2 角色对话链详细规范

### 2.1 通用对话格式

每个角色的输出遵循统一格式：

```markdown
### 【{emoji} {角色名}】— {角色英文名} 视角

**输入**: {上一角色的输出摘要}

**分析**:
{角色专业视角的分析过程}

**结论**:
{本角色的决策/输出}

**交接给**: {下一角色名}
---
```

### 2.2 R1 制作人（Producer）

**触发时机**: 收到用户的游戏需求描述时。

**输出模板**:

```markdown
### 【🎯 制作人】— Producer 视角

**用户原始需求**: {用户输入}

**需求澄清**:
- 游戏类型: {2D/3D/混合}
- 核心玩法: {一句话描述}
- 目标平台: {移动端/PC/全平台}
- 复杂度评估: {简单/中等/复杂}

**范围界定**:
- 必须实现: {功能列表}
- 可选功能: {功能列表}
- 明确排除: {不做的功能}

**风险识别**:
- {风险1}: {应对策略}
- {风险2}: {应对策略}

**脚手架选择**: {scaffold-2d / scaffold-2d-physics / scaffold-3d-scene / scaffold-3d-character}
**预估模块数**: {N} 个文件

**交接给**: R2 策划
```

**R1 引擎规则检查**:
- 确认游戏类型对应正确的脚手架（CLAUDE.md 规则 #2）
- 评估是否需要模块化（CLAUDE.md 规则 #13：>1500行必须拆分）

### 2.3 R2 策划（Designer）

**输入**: R1 的需求规格书。

**输出模板**:

```markdown
### 【📋 策划】— Designer 视角

**基于制作人的需求规格**:

**玩法设计**:
- 核心循环: {描述游戏循环}
- 操作方式: {键盘/鼠标/触屏}
- 胜负条件: {描述}

**数值框架**:
| 参数 | 初始值 | 说明 |
|------|--------|------|
| {参数名} | {值} | {说明} |

**用户体验流程**:
1. {启动 → 主菜单}
2. {主菜单 → 游戏}
3. {游戏 → 结算}

**UI 需求**:
- {UI元素列表}
- UI 方案: urhox-libs/UI 组件（规则 #10）

**交接给**: R3 架构师
```

**R2 引擎规则检查**:
- UI 方案必须使用 urhox-libs/UI（规则 #10）
- 长度单位使用米（规则 #0）
- 坐标系 Y-up 左手（规则 #0.5）

### 2.4 R3 架构师（Architect）

**输入**: R1 需求规格 + R2 策划文档。

**输出模板**:

```markdown
### 【🏗️ 架构师】— Architect 视角

**技术选型**:
- 脚手架: {具体脚手架文件}
- 物理引擎: {Box2D / Bullet / 无}
- 渲染方案: {3D PBR / NanoVG / 混合}
- UI 系统: urhox-libs/UI

**模块拆分**:
```
scripts/
├── main.lua          # 入口，场景初始化
├── GameConfig.lua    # 游戏配置常量
├── {Module1}.lua     # {模块1说明}
├── {Module2}.lua     # {模块2说明}
└── ...
```

**模块依赖关系**:
- main.lua → GameConfig, {Module1}, {Module2}
- {Module1} → GameConfig

**关键技术决策**:
1. {决策1}: {选择A vs 选择B} → 选择 {X}，因为 {理由}
2. {决策2}: ...

**交接给**: R4 程序员
```

**R3 引擎规则检查**:
- 代码放在 scripts/ 目录（规则 #1）
- 资源路径不加 assets/ 前缀（规则 #1.5）
- 程序化材质用 PBRNoTexture 系列（规则 #9.6）
- 3D 模型尺寸查文档或用 boundingBox（规则 #9.1）
- 第三人称相机用 ThirdPersonCamera 库（规则 #13.5）

### 2.5 R4 程序员（Programmer）

**输入**: R1-R3 的全部输出。

**行为**:
1. 按 R3 的模块拆分，逐文件编写代码
2. 基于选定的脚手架开始（不从零开始）
3. 代码遵循所有 CLAUDE.md 引擎规则
4. 首次交付添加日志（`print` 调试信息）

**输出**: 直接写入 `scripts/` 目录的代码文件。

**R4 引擎规则全面检查（编码时）**:
- Lua 数组索引从 1 开始（规则 #4）
- table.unpack 陷阱（规则 #4.5）
- eventData 访问用 GetInt/GetFloat（规则 #3）
- 枚举值不用数字（规则 #12）
- NanoVG 用 NanoVGRender 事件（规则 #6）
- NanoVG 文本先创建字体（规则 #7）
- 分辨率用 GetWidth/GetHeight/GetDPR（规则 #0.8）
- FPS/TPS 设置 MM_RELATIVE（规则 #9）
- 类型标注空值变量（规则 #11）

### 2.6 R5 审查员（Reviewer）

**输入**: R4 编写的代码文件。

**输出模板**:

```markdown
### 【🔍 审查员】— Reviewer 视角

**审查范围**: {文件列表}

**引擎规则合规检查**:
| 规则 | 状态 | 说明 |
|------|------|------|
| #0 长度单位(米) | ✅/❌ | {说明} |
| #1 代码在 scripts/ | ✅/❌ | {说明} |
| #4 数组索引从1 | ✅/❌ | {说明} |
| #6 NanoVGRender 事件 | ✅/N/A | {说明} |
| #10 UI 用 urhox-libs/UI | ✅/N/A | {说明} |
| #12 枚举不用数字 | ✅/❌ | {说明} |
| ... | ... | ... |

**代码质量**:
- {问题1}: {建议}
- {问题2}: {建议}

**判定**: {通过 / 需修改}

**若需修改**: → 打回 R4，附修改清单
**若通过**: → 交接给 R6 测试员
```

**R5 打回规则**:
- 发现引擎规则违反 → 必须打回 R4
- 发现逻辑错误 → 打回 R4
- 仅有代码风格建议 → 通过，建议附在报告中

### 2.7 R6 测试员（Tester）

**输入**: R5 审查通过的代码。

**行为**:
1. 调用 UrhoX MCP `build` 工具
2. 分析构建结果
3. 若失败 → 触发自我修正循环

**输出模板（成功）**:

```markdown
### 【🧪 测试员】— Tester 视角

**构建结果**: ✅ 成功

**验证项目**:
- [x] 构建无错误
- [x] 入口函数 Start() 存在
- [x] 资源引用路径正确

**交接给**: R7 终审员
```

**输出模板（失败，触发修正循环）**:

```markdown
### 【🧪 测试员】— Tester 视角

**构建结果**: ❌ 失败（第 {N}/3 轮修正）

**错误分析**:
- 错误类型: {语法错误/引用错误/逻辑错误}
- 错误位置: {文件:行号}
- 错误信息: {构建工具输出}

**修正建议**:
1. {具体修改建议1}
2. {具体修改建议2}

**→ 回退到 R4 程序员执行修正**
```

### 2.8 R7 终审员（Final Reviewer）

**输入**: R6 测试通过的项目。

**输出模板**:

```markdown
### 【✅ 终审员】— Final Reviewer 视角

**验收清单**:
- [x] 构建成功（零错误）
- [x] 核心功能完整（对照 R1 需求）
- [x] 引擎规则合规（对照 R5 报告）
- [x] 代码模块化合理
- [x] 首次交付包含日志

**交付摘要**:
- 文件数: {N}
- 总行数: {约 N 行}
- 脚手架: {使用的脚手架}
- 自我修正轮次: {0-3}

**项目状态**: completed ✅

**后续建议**:
1. {可优化的方向}
2. {可扩展的功能}
```

## §5 跨会话恢复

### 5.1 恢复流程

当新会话开始时，检测 `docs/factory-state.json` 是否存在：

```
新会话开始
  ↓
检查 docs/factory-state.json
  ↓
├─ 存在且 status \!= "completed"
│    ↓
│    读取状态 → 报告当前进度 → 从 current_role 继续
│
├─ 存在且 status == "completed"
│    ↓
│    报告已完成 → 询问是否开始新项目
│
└─ 不存在
     ↓
     全新项目 → 从 R1 开始
```

### 5.2 恢复时的输出

```markdown

| 项目 | 值 |
|------|------|

| 已完成 | {角色列表} |
| 修正轮次 | {N}/3 |

正在从 **{角色名}** 阶段继续...

## §6 与其他 Skill 的协作

### 6.1 协作矩阵

| 阶段 | 可调用的 Skill | 调用时机 |
|------|---------------|---------|
| R2 策划 | `@jrpg_jrpg-design` | 用户需求为 JRPG 类型时 |
| R3 架构 | `@zy_game-architect-v2` | 需要详细子系统设计时 |
| R3 架构 | `materials` | 涉及 PBR 材质选择时 |
| R3 架构 | `setup-fsm` | 涉及角色动画状态机时 |
| R4 编码 | `materials` | 创建程序化材质时 |
| R4 编码 | `audio-manager` | 添加音频系统时 |
| R5 审查 | `game-bug-checker` | 深度代码缺陷扫描时 |
| R6 测试 | UrhoX MCP build | **必须调用** |

### 6.2 与 game-mission-orchestrator 的边界

| 维度 | game-dev-factory | game-mission-orchestrator |
|------|-----------------|--------------------------|
| 核心模式 | 角色轮转对话链 | 里程碑阶段推进 |
| 推进方式 | R1→R2→...→R7 链式 | Phase 1→2→3→4 阶段式 |
| 修正机制 | 自我修正循环（R6↔R4） | 质量门禁 + 手动修复 |
| 状态文件 | docs/factory-state.json | docs/snapshot.json |
| 适用场景 | 单次功能开发 | 长期项目管理 |

两者可组合：`game-mission-orchestrator` 管理宏观阶段，每个阶段内部用 `game-dev-factory` 的角色链执行。

## §8 决策树

```
用户描述了游戏需求
  ↓
是否明确要求"多角色协作"/"工厂模式"/"虚拟团队"?
├─ 是 → 使用 game-dev-factory
└─ 否 ↓
     是否要求"制定开发计划"/"里程碑"/"阶段推进"?
     ├─ 是 → 建议 game-mission-orchestrator
     └─ 否 ↓
          是否复杂游戏（多系统、多模块）?
          ├─ 是 → 建议 game-dev-factory 或 @vintlin_team-dev
          └─ 否 → 直接开发（无需流程 skill）
