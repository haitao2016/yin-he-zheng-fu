# 角色对话模板扩展指南

> 本文档提供角色链中各角色的扩展模板和自定义指南。

---

## 1. 角色模板自定义规则

### 1.1 输出格式约定

每个角色输出 **必须** 包含以下结构：

```markdown
### 【{emoji} {角色名}】— {英文名} 视角

**输入**: {上一角色的输出摘要}
**分析**: {专业视角分析}
**结论**: {决策/输出}
**交接给**: {下一角色}
```

### 1.2 Emoji 映射表

| 角色 | Emoji | 固定，不可修改 |
|------|-------|---------------|
| R1 制作人 | 🎯 | 目标导向 |
| R2 策划 | 📋 | 规划设计 |
| R3 架构师 | 🏗️ | 结构搭建 |
| R4 程序员 | 💻 | 编码实现 |
| R5 审查员 | 🔍 | 检查审视 |
| R6 测试员 | 🧪 | 验证测试 |
| R7 终审员 | ✅ | 最终确认 |

---

## 2. 角色间交接规则

### 2.1 正向链（Normal Chain）

```
R1 → R2 → R3 → R4 → R5 → R6 → R7
```

每个角色只有在完成输出后才交接给下一角色。交接时必须明确标注"交接给"。

### 2.2 回退链（Kickback Chain）

| 发起者 | 目标 | 触发条件 |
|--------|------|---------|
| R5 审查员 | R4 程序员 | 发现引擎规则违反或逻辑错误 |
| R6 测试员 | R4 程序员 | 构建失败（自我修正循环） |

**回退格式**:

```markdown
### 【🔍 审查员】— 打回修改

**打回原因**: {具体问题}
**修改清单**:
1. {文件}:{行号} — {问题描述} → {修改建议}
2. ...

**→ 回退到 R4 程序员**
```

### 2.3 R5 打回判定标准

| 类别 | 严重度 | 处理 |
|------|--------|------|
| 引擎规则违反 | 🔴 阻断 | 必须打回 |
| 逻辑错误 | 🔴 阻断 | 必须打回 |
| 性能问题 | 🟡 警告 | 通过，附建议 |
| 代码风格 | 🟢 建议 | 通过，附建议 |

## 3. 角色输出的详细程度

### 3.1 简单项目（< 200 行代码）

- R1: 简化为 3-5 行需求确认
- R2: 简化为核心玩法 + 数值表
- R3: 简化为文件列表 + 脚手架选择
- R4-R7: 正常流程

### 3.2 中等项目（200-800 行代码）

- 所有角色正常输出

### 3.3 复杂项目（> 800 行代码）

- R1: 增加风险分析深度
- R2: 增加数值平衡分析
- R3: 增加模块依赖图
- R4: 分模块逐步实现
- R5-R7: 正常流程

---

## 4. 多轮迭代场景

当用户在项目完成后要求添加新功能时：

1. **不重新走全链**，从相关角色开始：
   - 小改动（UI调整、数值修改）→ 从 R4 开始
   - 新功能（新模块、新系统）→ 从 R2 开始
   - 架构变更（重构、拆分）→ 从 R3 开始

2. **更新 factory-state.json** 中的 `roles_completed` 为部分角色

3. **保留之前的 role_outputs** 作为上下文

---

## 5. 角色跳过规则

在以下情况下，部分角色可以简化或跳过：

| 条件 | 可简化的角色 |
|------|-------------|
| 用户提供了完整策划文档 | R2 简化为确认 |
| 用户指定了架构方案 | R3 简化为确认 |
| 修复 bug（非新功能） | R1-R3 跳过，从 R4 开始 |
| 用户要求快速模式 | R1-R3 合并为一个输出 |

**注意**: R4（程序员）、R6（测试员）**永远不可跳过**。
