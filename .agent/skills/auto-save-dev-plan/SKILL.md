# 开发计划自动保存 Skill

## 核心价值

解决以下痛点：
1. **计划丢失**：会话结束后，精心制定的开发计划随上下文消失
2. **重复劳动**：新会话需要重新制定相同的开发计划
3. **进度断裂**：无法从上次中断处继续开发
4. **版本混乱**：计划多次修改后，无法追溯历史版本
5. **完成无记录**：开发全部完成后，缺少结项归档和完成摘要
6. **缺少闭环**：完成或中断后没有总结回顾，也没有后续引导

## 触发机制

### 自动触发（Hook 模式）

当以下条件满足时，**自动执行对应流程**：

| 触发场景 | 触发时机 | 识别信号 | 执行流程 |
|---------|---------|---------|---------|
| `game-dev-planner` 规划完成 | Phase 4（DEV PLAN）完成 | 输出了完整的任务拆解列表 | Flow 1（保存） |
| `structured-dev-session` 规划完成 | Phase 1（ANALYZE）完成 | 输出了 Session 概要和执行计划 | Flow 1（保存） |
| `@vintlin_team-dev` 规划完成 | PLAN 阶段完成 | 输出了 task-registry 任务队列 | Flow 1（保存） |
| 自由对话中规划完成 | AI 生成了完整开发计划 | 对话中出现了结构化任务列表 | Flow 1（保存） |
| **开发全部完成** | **所有任务标记为 completed** | **TodoWrite 中无 pending/in_progress 任务** | **Flow 5（结项）→ Flow 6（总结引导）** |

**识别"开发计划已制定完成"的信号**：
- 对话中出现了**编号任务列表**（如 1. xxx 2. xxx ...）
- 对话中出现了**技术选型**或**架构决策**
- 对话中出现了**文件结构规划**
- TodoWrite 中写入了多个待办任务
- AI 说出类似"计划制定完成"、"接下来开始实现"的过渡语

**识别"开发计划已全部完成"的信号**：
- TodoWrite 中**所有任务**状态为 `completed`（无 pending 或 in_progress）
- AI 说出类似"所有任务已完成"、"开发完成"、"全部搞定"的总结语
- 上游 Skill 进入最终阶段（如 game-dev-planner 的 Phase 7 DELIVERY、team-dev 的 ARCHIVE）
- 用户确认"开发完成了"、"可以了"、"没其他要做的了"

### 手动触发

用户主动说出以下关键词时触发：
- "保存开发计划"、"保存计划"、"持久化计划"、"导出计划"
- "恢复开发计划"、"加载计划"、"继续上次的计划"
- "更新计划"、"计划变更"

## 存储规范

### 目录结构

```
docs/plans/
  index.md                              # 计划索引（所有计划的目录）
  {YYYY-MM-DD}_{plan-name}.plan.md      # 计划文件
  archive/                              # 历史版本归档
    {YYYY-MM-DD}_{plan-name}.v{N}.plan.md
```

### 文件命名规则

- 日期前缀：`YYYY-MM-DD` 格式（计划创建日期）
- 计划名：从项目名或用户描述中提取，使用小写中划线格式
- 后缀：`.plan.md`
- 示例：`2026-05-13_tower-defense-game.plan.md`

### 计划文件格式

每个计划文件遵循统一的模板格式（见 `references/templates/plan-template.md`），包含以下 sections：

```markdown
# {项目名} - 开发计划

> 创建时间: {YYYY-MM-DD HH:MM}
> 最后更新: {YYYY-MM-DD HH:MM}
> 来源: {触发来源，如 game-dev-planner / 自由对话}
> 状态: 进行中 | 已完成 | 已归档

## 1. 项目概述
简述项目目标、核心玩法、技术栈。

## 2. 需求摘要
用户的原始需求要点。

## 3. 技术选型
脚手架选择、关键库/组件、架构模式。

## 4. 任务拆解
使用状态标记的任务列表。

## 5. 文件结构
预期的代码文件组织。

## 6. 风险与注意事项
已知的技术风险、性能关注点。

## 7. 验收标准
什么条件下认为开发完成。

---
## 修订记录
| 版本 | 日期 | 变更内容 |
|------|------|---------|
| v1   | {date} | 初始计划 |
```

### 索引文件格式 (index.md)

```markdown
# 开发计划索引

| 计划名 | 创建日期 | 状态 | 文件 |
|--------|---------|------|------|
| {name} | {date}  | {status} | [{filename}](./{filename}) |
```

## 执行流程

### Flow 1: 自动保存（规划阶段完成后）

```
检测到开发计划制定完成信号
  ↓
1. 提取计划内容
   - 从对话上下文中提取：项目概述、需求、技术选型、任务列表、文件结构
   - 从 TodoWrite 中提取当前任务状态
  ↓
2. 生成计划文件
   - 确定文件名：{YYYY-MM-DD}_{plan-name}.plan.md
   - 按模板填充内容
   - 写入 docs/plans/ 目录
  ↓
3. 更新索引
   - 读取或创建 docs/plans/index.md
   - 追加新计划条目
  ↓
4. 通知用户
   - 简要告知已保存，显示文件路径
   - 不中断当前工作流
```

### Flow 2: 手动保存

```
用户说"保存开发计划"
  ↓
1. 收集当前对话中的计划信息
   - 如果已有计划文件 → Flow 4（更新）
   - 如果没有 → 继续
  ↓
2-4. 同 Flow 1 的步骤 2-4
```

### Flow 3: 会话恢复

```
用户说"恢复开发计划" / "继续上次的计划" / "加载计划"
  ↓
1. 读取 docs/plans/index.md
   - 如果不存在 → 告知没有已保存的计划
  ↓
2. 展示可用计划列表
   - 如果只有一个 → 直接加载
   - 如果有多个 → 让用户选择
  ↓
3. 读取计划文件内容
  ↓
4. 恢复上下文
   - 将计划内容注入当前会话上下文
   - 根据任务状态标记，确定从哪里继续
   - 将未完成的任务写入 TodoWrite
  ↓
5. 告知用户恢复结果
   - 显示已完成/未完成的任务统计
   - 提示从哪个任务继续
```

### Flow 4: 计划更新（开发过程中）

```
检测到计划变更（单个任务完成、需求变更、新增任务等）
  或用户说"更新计划"
  ↓
1. 查找现有计划文件
   - 按日期和名称匹配最新的计划文件
  ↓
2. 归档旧版本
   - 将当前版本复制到 archive/ 目录
   - 命名为 {原名}.v{N}.plan.md（N 递增）
  ↓
3. 更新计划文件
   - 更新任务状态标记（⬜ → 🔄 → ✅）
   - 更新"最后更新"时间
   - 追加修订记录
  ↓
4. 更新索引文件中的状态
```

### Flow 5: 开发完成自动结项

```
检测到所有任务已完成
  ↓
1. 确认完成状态
   - 检查 TodoWrite：所有任务均为 completed
   - 检查对话上下文：AI 输出了完成总结或用户确认完成
   - 如果仍有未完成任务 → 不触发，等待
  ↓
2. 归档当前版本
   - 将当前计划文件复制到 archive/ 目录
  ↓
3. 更新计划文件
   - 将所有任务标记更新为 ✅
   - 将文件头部状态改为"已完成"：
     > 状态: 已完成
   - 更新"最后更新"时间
   - 追加完成摘要（见下方格式）
   - 追加修订记录：标注"开发完成，结项归档"
  ↓
4. 更新索引文件
   - 将对应计划条目状态改为"已完成"
  ↓
5. 进入 Flow 6（总结与后续引导）
```

**Flow 6 的触发时机汇总**：

| 触发场景 | 入口 | 走哪条分支 |
|---------|------|-----------|
| Flow 5 结项完成后 | 自动衔接 | 2A（已完成） |
| 会话恢复（Flow 3）后检测到计划已完成 | 自动衔接 | 2A（已完成） |
| 会话恢复（Flow 3）后检测到计划未完成 | 自动衔接 | 2B（未完成） |
| 用户中途说"不做了"/"先暂停" | 手动触发 | 2B（未完成） |

**全部任务列表输出规则**：

1. **按照计划文件中的原始编号顺序列出所有任务**，不跳过任何一个
2. 每个任务前使用对应的状态标记（✅/⬜/🔄/❌/⏸️）
3. 对非"已完成"状态的任务，在描述后追加括号说明（如"进行中"、"已取消"、"暂停"）
4. 已完成的任务不追加额外说明
5. 如果某个任务有子任务，子任务也需列出并缩进显示

**完成摘要格式**（Flow 5 步骤 3 追加到计划文件修订记录之前）：

```markdown

- **完成时间**: {YYYY-MM-DD HH:MM}
- **总任务数**: {N}
- **完成任务**: {completed} (xx%)
- **取消任务**: {cancelled}

✅ 1. {任务1描述}
✅ 2. {任务2描述}
✅ 3. {任务3描述}

### 关键产出

- `scripts/{module}.lua` — {模块说明}
- ...

### 回顾
- **亮点**: {做得好的方面}
- **问题**: {开发中解决的关键问题}
- **可改进**: {后续优化方向}
```

## 任务状态标记

在计划文件的任务列表中使用统一的状态标记：

| 标记 | 含义 | 说明 |
|------|------|------|
| ⬜ | 待做 | 尚未开始 |
| 🔄 | 进行中 | 当前正在处理 |
| ✅ | 已完成 | 已成功完成 |
| ❌ | 已取消 | 决定不做了 |
| ⏸️ | 暂停 | 临时搁置，后续恢复 |

## Skill 协作

| 关系 | Skill | 协作方式 |
|------|-------|---------|
| 上游 | `game-dev-planner` | Phase 4 完成 → Flow 1；Phase 7 完成 → Flow 5 → Flow 6 |
| 上游 | `structured-dev-session` | Phase 1 完成 → Flow 1；Phase 7 完成 → Flow 5 → Flow 6 |
| 上游 | `@vintlin_team-dev` | PLAN 完成 → Flow 1；ARCHIVE 完成 → Flow 5 → Flow 6 |
| 互补 | `game-dev-project-tracker` | 追踪更细粒度的特性进度 |
| 互补 | `@Huiyu-Skill_memory-system` | 跨会话记忆与计划持久化互补 |
| 下游 | `@org_git-save` | 保存/结项后可触发 git 提交 |

## 注意事项

1. **不中断工作流**：自动保存和结项都是静默操作，仅用简短通知告知用户，不打断当前开发节奏
2. **幂等性**：多次保存同一计划不会产生重复文件，而是更新现有文件
3. **最小侵入**：只创建 `docs/plans/` 目录，不修改项目的其他文件结构
4. **格式兼容**：计划文件是标准 Markdown，人类可直接阅读和编辑
5. **结项不可逆**：Flow 5 标记"已完成"后，如需重新开启，应创建新的计划文件而非修改已完成的文件
6. **总结引导不强迫**：Flow 6 的询问只问一次，用户拒绝或转移话题后不再追问
7. **暂停可恢复**：用户暂停后，计划文件保留所有进度状态，下次恢复时自动从中断点继续
8. **全部任务必列**：Flow 6 的总结报告和进度总结中，必须列出计划中的每一个任务并标记状态，不得省略
