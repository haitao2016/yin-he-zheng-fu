# 质量门禁清单

Game Forge 在每个阶段完成后执行质量门禁审查。本文档定义了每个门禁的详细审查标准。

---

## REVIEW-1: 概念阶段审查

### 触发条件
Stage 1 所有必填章节完成后自动触发。

### 审查清单

#### 1.1 完整性检查
- [ ] 所有必填章节已完成（根据游戏类型判断）
- [ ] 每章超过最低字数要求（核心章节 ≥500 字，其他 ≥200 字）
- [ ] 每章至少分配了 1 个 ID

#### 1.2 核心一致性
- [ ] ch02(核心玩法)与 ch03(游戏循环)的描述一致
- [ ] ch05(难度)与 ch06(新手引导)的节奏不矛盾
- [ ] ch09(美术)与 ch10(UI)的风格统一
- [ ] ch11(技术)的选型支持 ch02(核心玩法)的需求

#### 1.3 UrhoX 可行性
- [ ] ch11 选择了合适的脚手架
- [ ] 渲染方案与游戏类型匹配（2D 自绘 → NanoVG，3D → PBR）
- [ ] UI 方案使用 urhox-libs/UI（不使用原生 UI）
- [ ] 物理引擎选择正确（2D → Box2D，3D → Bullet）
- [ ] 多人模式（如有）的架构方案可行

#### 1.4 深度检查
- [ ] ch02 包含 30 秒体验描述
- [ ] ch02 包含操控方式映射（移动端+PC）
- [ ] ch03 包含至少 2 层游戏循环
- [ ] ch09 包含具体色彩方案（HEX 值）
- [ ] ch10 包含界面跳转流程
- [ ] ch11 包含代码组织方案

#### 1.5 交叉引用
- [ ] 所有章节间的引用 ID 在注册表中存在
- [ ] 无悬空引用（引用了不存在的 ID）
- [ ] 无孤立元素（分配了 ID 但未被任何章节引用）

### 输出
- ✅ **通过**: 进入 Stage 2
- ⚠️ **有建议**: 列出改进建议，用户确认后进入 Stage 2
- ❌ **阻断**: 列出必须修复的问题，修复后重新审查

---

## REVIEW-2: 系统设计审查

### 触发条件
所有识别的游戏系统 SYS-*.md 完成后触发。

### 审查清单

#### 2.1 完整性检查
- [ ] ch02(核心玩法)涉及的所有功能都有对应的系统
- [ ] 每个系统有明确的职责描述
- [ ] 每个系统有接口定义（输入/输出）
- [ ] 每个系统有 UrhoX 实现方案
- [ ] CONTENT-RHYTHM.md 已完成

#### 2.2 系统依赖
- [ ] 系统间无循环依赖
- [ ] 依赖关系已在 SYS-*.md 中明确标注
- [ ] 核心系统（SYS-CORE）不依赖可选系统

#### 2.3 概念覆盖
- [ ] Stage 1 的每个 C-* ID 至少被一个 SYS-* 引用
- [ ] 无概念元素遗漏（未被任何系统覆盖）

#### 2.4 UrhoX 映射
- [ ] 每个系统映射到具体的 scripts/ 文件
- [ ] 文件组织符合模块化规范（预计超 1000 行的需拆分）
- [ ] 使用的 UrhoX API/组件在引擎中存在
- [ ] 输入系统使用 InputManager 跨平台适配

#### 2.5 数据需求
- [ ] 每个系统标注了需要的配置数据
- [ ] 数据需求可映射为 Lua 表结构

### 输出
- ✅ / ⚠️ / ❌（同 REVIEW-1）

---

## REVIEW-3: 数据与平衡审查

### 触发条件
Stage 3A(数据模式) + Stage 3B(数值平衡) 完成后触发。

### 审查清单

#### 3.1 数据完整性
- [ ] 所有 SYS-* 标注的数据需求都有对应的 TBL-*
- [ ] 每张表有主键定义
- [ ] 外键关系有效（引用的表和字段存在）
- [ ] 枚举定义完整（所有引用的枚举值已定义）

#### 3.2 Lua 配置验证
- [ ] scripts/config/ 中的 Lua 文件语法正确
- [ ] 数据类型与表定义一致
- [ ] 无空值或缺失的必填字段
- [ ] 可通过 UrhoX 构建工具验证

#### 3.3 数值平衡
- [ ] 难度曲线有明确的数学公式
- [ ] 经济模型有收支平衡分析
- [ ] 成长曲线不会过早饱和或过度膨胀
- [ ] 关键数值有合理的上下限

#### 3.4 UrhoX 单位一致性
- [ ] 速度单位：米/秒（m/s）
- [ ] 距离单位：米（m）
- [ ] 角色高度：1.5-2.0 米范围
- [ ] 重力加速度：-9.81 m/s²
- [ ] 跳跃初速度：合理范围（通常 5-10 m/s）

#### 3.5 追溯完整性
- [ ] 所有 TBL-* 和 BAL-* ID 已注册
- [ ] 从 C-* → SYS-* → TBL-*/BAL-* 的引用链完整
- [ ] ID 注册表与实际文档内容同步

### 输出
- ✅ / ⚠️ / ❌（同 REVIEW-1）

---

## FINAL-REVIEW: 最终审查

### 触发条件
Stage 4(制作规格) 完成后触发。

### 审查清单

#### 4.1 制作规格完整性
- [ ] ART-SPEC.md 包含完整的资源清单
- [ ] ART-SPEC.md 包含 AI 图像生成提示词
- [ ] UI-SPEC.md 包含完整的界面布局描述
- [ ] UI-SPEC.md 使用 urhox-libs/UI 组件
- [ ] TECH-SPEC.md 包含项目目录结构
- [ ] TECH-SPEC.md 包含构建配置

#### 4.2 全链路追溯
- [ ] C-* → SYS-* → TBL-*/BAL-* → PRD-* 完整链路
- [ ] 无悬空 ID（注册但未使用）
- [ ] 无未注册引用

#### 4.3 实施可行性
- [ ] TECH-SPEC 的脚手架与 ch11 一致
- [ ] 代码结构符合 UrhoX 模块化规范
- [ ] 所有引用的 UrhoX API 存在
- [ ] 构建配置完整（入口文件、多人模式设置等）

#### 4.4 交付物清单
- [ ] docs/gdd/ 目录结构完整
- [ ] scripts/config/ Lua 配置文件可用
- [ ] 所有 REVIEW-*.md 已生成
- [ ] STATE.md 状态为"全部完成"

### 通过标准
- 全部 ✅：设计阶段完成，可进入开发阶段
- 存在 ⚠️：列出建议，用户确认后标记完成
- 存在 ❌：必须修复后重新审查

## 门禁执行流程

```
阶段完成
  ↓
读取该阶段所有文档
  ↓
逐项执行审查清单
  ↓
生成 REVIEW-N.md
  ↓
├─ 全部通过 → 更新 STATE.md，解锁下一阶段
├─ 有建议 → 展示给用户，确认后解锁
└─ 有阻断 → 展示必修项，用户修复后重新审查
```

### 审查文件模板

```markdown
# REVIEW-N: [阶段名] 审查

## 审查时间: [日期]
## 审查结果: ✅ 通过 / ⚠️ 有建议 / ❌ 阻断

### 完整性 (X/Y 通过)
- [x] 检查项 1
- [ ] 检查项 2 — 💡 建议: ...
- [\!] 检查项 3 — 🚨 阻断: ...

### 一致性 (X/Y 通过)
...

### UrhoX 可行性 (X/Y 通过)
...

### 总结
- 通过项: X
- 建议项: Y
- 阻断项: Z

### 后续行动
1. [需要修复的项目]
2. [需要补充的内容]
```

## 审查严格度配置

根据游戏复杂度调整审查严格度：

| 复杂度 | 最低章节字数 | 循环层数要求 | 系统数量要求 |
|--------|------------|------------|------------|
| 简单(casual) | 150 字 | 2 层 | 3+ |
| 中等(puzzle/idle) | 200 字 | 2 层 | 4+ |
| 复杂(rpg/action/strategy) | 300 字 | 3 层 | 6+ |
