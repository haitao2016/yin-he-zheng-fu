# Game Studio Review — 多角色专家评审系统

> **灵感来源**: [game-studio-ai](https://github.com/miharbi/game-studio-ai) — 多 Agent 游戏工作室流水线

> 每位专家从独立的专业视角审查同一份代码/设计，
> 发现单一视角难以察觉的跨领域问题。

## 1. 概述与定位

### 1.1 这个 Skill 做什么

Game Studio Review 模拟一个 6 人专家团队对 UrhoX Lua 游戏项目进行结构化评审：

| 角色 | 关注焦点 | 对标 game-studio-ai |
|------|---------|-------------------|
| **创意总监** (Creative Director) | 游戏体验、玩法一致性、交互设计 | Tier 1: creative_director |
| **技术总监** (Technical Director) | 架构质量、引擎规则合规、代码健壮性 | Tier 1: technical_director |
| **制作人** (Producer) | 项目完整度、交付就绪、范围控制 | Tier 1: producer |
| **美术总监** (Art Director) | UI/UX 质量、视觉一致性、布局适配 | Tier 2: art_director |
| **QA 主管** (QA Lead) | 缺陷检测、边界条件、异常处理 | Tier 2: qa_lead |
| **性能工程师** (Performance Engineer) | 性能瓶颈、内存管理、帧率优化 | Tier 3: performance specialist |

### 1.2 不做什么

| 不做 | 应该用 |
|------|--------|
| 自动修复代码 | `game-bug-checker` Skill |
| 多任务编排执行 | `subagent-task-orchestrator` Skill |
| 完整开发流程管控 | `super-game-dev` / `structured-dev-session` Skill |
| 架构设计规划 | `@zy_game-architect-v2` Skill |
| 性能 Profiling 工具 | 直接使用引擎调试功能 |

### 1.3 与 game-studio-ai 的映射

```
game-studio-ai 原始架构          本 Skill 适配
─────────────────────────       ─────────────────────
28 个独立 Agent                → 6 个专家角色（精简合并）
3 层级结构                     → 保留（Tier1 战略/Tier2 执行/Tier3 专项）
YAML 计划模板                  → Markdown 评审模板
human_review 门控              → 用户确认门控（评审结果需用户确认）
引擎检测(Godot/Unity/Unreal)   → 固定为 UrhoX 引擎规则集
输出验证器                     → 严重等级分类 + 行动清单
Agent 输出合并器               → 汇总报告 + 优先级排序
```

### 2.2 不重叠声明

| 现有 Skill | 它做什么 | 本 Skill 不做什么 |
|-----------|---------|-----------------|
| `game-bug-checker` | 17 类 BUG 自动扫描修复 | 不自动修复，只输出评审意见 |
| `super-game-dev` | 8 阶段治理流水线 | 不管控开发流程阶段 |
| `structured-dev-session` | 7 阶段单功能开发 | 不执行开发步骤 |

| `auto-workflow` | 检测重复模式生成代码 | 不生成代码 |

**核心区分**: 本 Skill 是 **只读分析** — 输入代码，输出结构化评审报告，不修改任何文件。

## 3. 六大专家角色定义

### 3.1 创意总监 (Creative Director)

**层级**: Tier 1（战略层）
**职责**: 审查游戏体验的整体一致性

**评审视角**:
- 玩法循环是否清晰（核心循环 → 目标 → 反馈 → 奖励）
- 交互设计是否直觉化（玩家无需教程即可理解基本操作）
- 游戏节奏是否合理（难度曲线、奖励频率）
- 视觉反馈是否充足（操作 → 即时响应）
- 主题/风格是否一致（美术、音效、文案统一性）

**检查项示例**:
```
[CD-001] 玩法循环完整性
  - 是否存在明确的核心游戏循环？
  - 胜负条件是否清晰？
  - 玩家目标是否可感知？

[CD-002] 交互反馈
  - 按键/点击是否有即时视觉或音频反馈？
  - 状态变化（血量、分数）是否有动画过渡？
  - 错误操作是否有提示？

[CD-003] 节奏控制
  - 难度是否随进度递增？
  - 是否有休息/过渡节点？
  - 奖励频率是否保持玩家动力？
```

### 3.2 技术总监 (Technical Director)

**层级**: Tier 1（战略层）
**职责**: 审查代码质量与引擎规则合规性

**评审视角**:
- UrhoX 引擎规则合规（14 条核心规则逐一检查）
- 代码架构合理性（模块化、职责分离）
- API 使用正确性（eventData、坐标系、枚举值）
- 错误处理覆盖（nil 检查、边界保护）
- 脚手架使用规范（是否基于标准脚手架）

**检查项示例**:
```
[TD-001] 引擎规则合规性
  □ 规则#0  长度单位使用米(meter)
  □ 规则#0.5 坐标系正确(Y-up 左手)
  □ 规则#0.8 分辨率使用 GetWidth/GetHeight/GetDPR
  □ 规则#1  代码在 scripts/ 目录
  □ 规则#2  基于标准脚手架
  □ 规则#3  eventData 使用 ["Key"]:GetType() 格式
  □ 规则#4  数组索引从 1 开始
  □ 规则#4.5 table.unpack 位置正确
  □ 规则#6  NanoVG 使用 NanoVGRender 事件
  □ 规则#7  NanoVG 字体只创建一次
  □ 规则#9  FPS/TPS 设置 MM_RELATIVE
  □ 规则#9.1 模型尺寸不猜测
  □ 规则#10 使用新 UI 系统 urhox-libs/UI
  □ 规则#12 使用枚举值不用数字

[TD-002] 代码架构
  □ 单文件 < 1500 行
  □ 函数职责单一
  □ 全局变量最小化
  □ 依赖引用规范

[TD-003] API 正确性
  □ eventData 访问方式正确
  □ require 路径正确
  □ 资源路径不含目录前缀
  □ 物理单位一致(米)
```

**引擎规则来源**: `engine-docs/principles.md`、`engine-docs/lua-scripting-guide.md`

### 3.3 制作人 (Producer)

**层级**: Tier 1（战略层）
**职责**: 审查项目完整度与交付就绪性

**评审视角**:
- 功能完整性（承诺的功能是否全部实现）
- 用户体验闭环（开始 → 游戏 → 结束 → 重开）
- 缺失功能识别（常见遗漏：暂停、重开、音量控制）
- 构建就绪性（是否能成功调用 build 工具）
- 多人模式一致性（检查 settings.json 与代码实现是否匹配）

**检查项示例**:
```
[PR-001] 功能完整性
  □ 游戏有明确的开始状态
  □ 游戏有明确的结束条件
  □ 支持重新开始
  □ 有暂停功能（如需要）
  □ 有基本设置（音量等）

[PR-002] 交付清单
  □ 代码能通过 build
  □ 所有引用的资源文件存在
  □ 无 TODO/FIXME 遗留
  □ 调试日志已清理（或标注保留）
  □ 版本号/标题已设置

[PR-003] 多人模式一致性（仅多人游戏）
  □ settings.json multiplayer.enabled 与代码一致
  □ 客户端/服务端入口文件对应
  □ 网络事件已注册
```

**引擎规则来源**: `engine-docs/principles.md`（规则#4 日志要求）、`engine-docs/lua-scripting-guide.md`（规则#14 多人模式判断）

### 3.4 美术总监 (Art Director)

**层级**: Tier 2（执行层）
**职责**: 审查视觉质量与 UI/UX 规范

**评审视角**:
- UI 系统选择正确性（必须使用 urhox-libs/UI）
- 布局适配（DPI 处理、响应式设计）
- 视觉一致性（颜色方案、字体选择、间距规范）
- 材质使用规范（PBR 材质、Technique 路径正确）
- NanoVG 使用场景是否恰当（仅自定义图形，非通用 UI）

**检查项示例**:
```
[AD-001] UI 系统规范
  □ 使用 urhox-libs/UI 而非原生 UI
  □ UI.Init() → 构建树 → UI.SetRoot() 三步完整
  □ 使用 UI.Scale.DEFAULT
  □ 未遗漏 UI.SetRoot() 调用

[AD-002] 布局适配
  □ 使用 Flexbox 布局而非硬编码坐标
  □ 使用 GetWidth/GetHeight/GetDPR 获取分辨率
  □ 未使用 graphics:SetMode()
  □ 容器设置 flexShrink 防止溢出

[AD-003] 视觉一致性
  □ 颜色使用一致（无随意硬编码颜色值）
  □ 字体家族统一
  □ 间距/边距规律
  □ 图标/装饰风格统一

[AD-004] 材质规范（3D 项目）
  □ 程序化材质使用 PBRNoTexture 系列
  □ Technique 路径正确（非猜测）
  □ 模型尺寸通过 boundingBox 获取
```

**引擎规则来源**: `engine-docs/recipes/ui.md`、`.claude/skills/materials/SKILL.md`、`engine-docs/built-in-models.md`

### 3.5 QA 主管 (QA Lead)

**层级**: Tier 2（执行层）
**职责**: 审查缺陷风险与边界处理

**评审视角**:
- 空值/nil 安全（变量访问前是否检查）
- 数组边界安全（索引从 1 开始、越界保护）
- 输入验证（用户输入的范围检查）
- 异常路径处理（资源加载失败、网络断开等）
- 状态机完整性（所有状态是否有转换定义）
- 平台兼容性（Web 特殊行为：Pointer Lock ESC 退出等）

**检查项示例**:
```
[QA-001] 空值安全
  □ 资源加载后检查 nil (cache:GetResource 返回值)
  □ 节点查找后检查 nil (scene:GetChild 返回值)
  □ 组件获取后检查 nil (node:GetComponent 返回值)
  □ 网络回调中检查连接状态

[QA-002] 数组安全
  □ 索引从 1 开始（非 0）
  □ 循环使用 for i = 1, #arr
  □ 随机索引 math.random(1, #arr)
  □ 索引计算使用 math.max(1, index)

[QA-003] 输入验证
  □ 鼠标按钮使用枚举 MOUSEB_LEFT
  □ 键盘按键使用枚举 KEY_SPACE
  □ 数值输入有范围限制
  □ 字符串输入有长度限制

[QA-004] 异常路径
  □ 资源缺失时有降级方案或错误提示
  □ 物理世界 nil 检查（场景可能未初始化）
  □ 定时器/回调在场景销毁后不执行
  □ 除零保护（除数可能为 0 的地方）

[QA-005] 平台兼容
  □ Web 平台 Pointer Lock ESC 退出处理
  □ 触屏输入兼容（如果需要支持移动端）
  □ DPI 差异处理（1x vs 2x vs 3x）
```

**引擎规则来源**: `engine-docs/lua-scripting-guide.md`（关键规则#2 eventData、#4 数组索引、#12 枚举值）、`engine-docs/api/input.md`

### 3.6 性能工程师 (Performance Engineer)

**层级**: Tier 3（专项层）
**职责**: 审查性能瓶颈与资源管理

**评审视角**:
- 每帧分配（Update 中创建对象、字符串拼接）
- 资源缓存（重复 GetResource 调用）
- NanoVG 资源泄漏（字体/图片每帧创建）
- 物理性能（碰撞体数量、射线检测频率）
- 渲染负载（DrawCall 数量、透明物体排序）
- 内存模式（大表频繁创建销毁、闭包累积）

**检查项示例**:
```
[PE-001] 每帧热路径
  □ Update/HandleUpdate 中无 table 创建
  □ Update 中无字符串拼接（.. 操作符）
  □ Update 中无 cache:GetResource 调用
  □ Update 中无 Scene/Node 创建（除非必要）
  □ 高频事件回调中无重计算

[PE-002] 资源管理
  □ 纹理/模型/声音在 Start 中预加载
  □ nvgCreateFont 只调用一次（非每帧）
  □ nvgCreateImage 只调用一次（非每帧）
  □ 材质对象缓存复用

[PE-003] 物理性能
  □ 静态物体使用 BT_STATIC（非 BT_DYNAMIC）
  □ 射线检测频率合理（非每帧多次）
  □ 碰撞掩码正确设置（减少不必要碰撞检测）
  □ 休眠物体不被频繁唤醒

[PE-004] 渲染优化
  □ 相似材质合批（减少 DrawCall）
  □ 远处物体使用 LOD 或裁剪
  □ 透明物体数量合理
  □ 未使用的灯光/阴影关闭

[PE-005] 内存模式
  □ 大数组/表在不需要时置 nil
  □ 闭包未累积引用（事件解绑）
  □ 临时对象使用对象池模式
  □ 定时器清理（避免泄漏回调）
```

**引擎规则来源**: `engine-docs/lua-scripting-guide.md`（性能相关）、`engine-docs/recipes/ui.md`（NanoVG 资源管理）

### 4.2 执行步骤详解

**Phase 0: 扫描与准备**

AI 执行以下操作：

1. 使用 Glob 扫描 `scripts/**/*.lua` 获取所有 Lua 文件
2. 使用 Read 读取每个文件内容
3. 使用 Read 读取 `.project/settings.json`（如存在）
4. 统计：
   - 总行数
   - 文件数
   - require 依赖图
   - 是否使用 NanoVG
   - 是否使用 UI 组件
   - 是否使用物理系统（2D/3D）
   - 是否使用网络功能
5. 输出项目概览摘要

**Phase 1-3: 角色评审**

AI 依次切换视角，以每个专家角色的身份审查代码：
- 每个角色输出独立的评审报告（使用第 6 节的模板）
- 每个发现标注严重等级（使用第 7 节的定义）
- 每个发现引用具体代码位置（文件:行号）

**Phase 4: 汇总**

AI 合并所有角色的发现：
- 去除重复发现（同一问题被多个角色报告）
- 按严重等级排序
- 生成最终行动清单（使用第 8 节的模板）

## 7. 严重等级定义

### 7.1 四级分类

| 等级 | 含义 | 处理要求 |
|------|------|---------|
| **P0** | **阻塞** — 导致崩溃/无法运行 | 必须立即修复 |
| **P1** | **严重** — 功能错误/规则违反 | 发布前必须修复 |
| **P2** | **改进** — 代码质量/可维护性 | 建议修复 |
| **P3** | **建议** — 体验优化/锦上添花 | 可选修复 |

### 7.2 典型 P0 场景

以下问题总是 P0：

```
- eventData 访问方式错误 → 运行时崩溃
- 数组索引从 0 开始 → nil 访问崩溃
- NanoVG 未使用 NanoVGRender 事件 → 画面空白
- nvgCreateFont 每帧调用 → 显存泄漏直到崩溃
- SetMode() 调用 → 无效，分辨率不正确
- 使用数字常量判断鼠标按钮 → 逻辑错误
- 资源路径包含目录前缀 → 资源加载失败
- Unicode 使用 \uXXXX → 语法错误
- UI.SetRoot() 遗漏 → UI 不显示
- settings.json multiplayer 与代码不一致 → 模式错误
```

### 7.3 等级升降规则

```
默认等级可根据上下文调整:

升级:
  - 问题出现在核心路径上 → 升一级
  - 同类问题出现 3+ 次 → 升一级
  - 影响多人模式同步 → 升一级

降级:
  - 问题在注释/文档中 → 降一级
  - 问题有运行时保护（pcall等）→ 降一级
  - 仅影响调试/开发阶段 → 降一级
```

### 12.1 高频引用规则

| 规则 | 说明 | 典型错误 | 正确写法 |
