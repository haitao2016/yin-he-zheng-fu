# Super Game Dev — UrhoX 游戏开发治理流水线

将完整游戏开发过程编排为 **8 阶段门控流水线**，以文档驱动开发，
用强制门禁、质量评分、专家角色和返工机制确保交付质量。

> **定位差异**：
> - `game-dev-planner`: 前置需求规划，无门禁暂停和质量评分
> - `structured-dev-session`: 单任务执行流水线，无三文档体系
> - `game-creation-workflow`: 4 阶段快速流程，无强制暂停和返工
> - `game-review-improve`: 事后质量审计，不参与开发过程
> - **本 Skill**: 贯穿全流程的**治理层**——文档驱动 + 门禁暂停 + 量化评分 + 专家角色

---

## 1. 流水线总览

```
┌─────────────────────────────────────────────────────────────────┐
│  Stage 1  RESEARCH       调研分析：平台能力、同类游戏、技术可行性  │
│  Stage 2  GAME DOCS      三文档生成：GDD + 架构文档 + UI/UX 文档  │
│  Stage 3  CONFIRM ⛔     强制门禁：用户确认三文档后才能继续        │
│  Stage 4  TASK PLAN      任务拆解：按模块拆分实现任务 + 依赖排序   │
│  Stage 5  IMPLEMENT      分模块实现：按任务计划逐个编码           │
│  Stage 6  PREVIEW ⛔     强制门禁：用户预览效果并确认              │
│  Stage 7  QUALITY        质量检查：评分 ≥ 80 分才可继续           │
│  Stage 8  DELIVER        交付打包：构建 + 发布材料准备             │
└─────────────────────────────────────────────────────────────────┘

⛔ = 强制暂停点，必须等待用户确认才能继续
```

### 阶段状态机

每个阶段有三个状态：

| 状态 | 含义 |
|------|------|
| ⬜ 未开始 | 尚未进入该阶段 |
| 🔵 进行中 | 当前正在执行 |
| ✅ 已完成 | 通过准出标准 |

---

## 2. 各阶段详细说明

### Stage 1: RESEARCH — 调研分析

**目标**: 在编码前充分理解需求和技术约束。

**准入**: 用户提供了游戏需求描述

**执行内容**:

1. **平台能力调研**
   - UrhoX 引擎能力边界（2D/3D、物理、UI、音频、网络）
   - 可用的 urhox-libs 组件
   - 参考 `engine-docs/` 确认 API 可行性

2. **同类游戏分析**
   - 核心玩法机制拆解
   - 技术实现难点预判
   - 参考 `examples/` 中的相似示例

3. **技术可行性评估**
   - 哪些功能引擎原生支持
   - 哪些需要自定义实现
   - 性能风险点（粒子数量、物理体数量等）

**产出**: `docs/research.md`（调研报告）

**准出**: 调研报告完成，关键技术风险已识别

---

### Stage 2: GAME DOCS — 三文档生成

**目标**: 生成三份核心设计文档，作为后续开发的唯一真相源。

**准入**: Stage 1 调研报告完成

**三文档体系**:

| 文档 | 内容 | 文件 |
|------|------|------|
| **GDD** (游戏设计文档) | 核心玩法、规则、胜负条件、数值框架 | `docs/gdd.md` |
| **架构文档** | 模块划分、数据流、代码结构、API 选型 | `docs/architecture.md` |
| **UI/UX 文档** | 界面布局、交互流程、视觉风格、控件选型 | `docs/uiux.md` |

**GDD 必须包含**:
```markdown
# 游戏设计文档 — [游戏名]

## 核心玩法
- 玩家做什么？
- 核心循环是什么？

## 规则与机制
- 胜负条件
- 得分/进度系统
- 难度曲线

## 数值框架
- 关键数值（速度、生命值、伤害等）
- 数值平衡考虑

## 内容规划
- 关卡/场景列表
- 角色/道具列表
```

**架构文档必须包含**:
```markdown
# 架构文档 — [游戏名]

## 技术选型
- 脚手架: scaffold-2d / scaffold-3d-character / ...
- UI 方案: urhox-libs/UI
- 物理引擎: Box2D / 3D Physics

## 模块划分
scripts/
├── main.lua          # 入口
├── game/             # 核心玩法
├── ui/               # UI 界面
└── utils/            # 工具模块

## 数据流
[描述模块间的数据流转]

## UrhoX API 依赖
[列出使用的主要 API]
```

**UI/UX 文档必须包含**:
```markdown
# UI/UX 文档 — [游戏名]

## 界面列表
- 主菜单
- 游戏 HUD
- 暂停菜单
- 结算界面

## 交互流程
[描述界面间的跳转逻辑]

## 视觉风格
- 配色方案
- 字体选择
- UI 组件选型（基于 urhox-libs/UI）

## 控件布局
[每个界面的控件布局草图描述]
```

**准出**: 三份文档全部完成

---

### Stage 3: CONFIRM ⛔ — 文档确认门禁

**目标**: 强制暂停，等待用户确认三文档。

**这是一个强制暂停点**。AI 必须：

1. 向用户展示三文档的核心摘要
2. 明确询问："三份设计文档已完成，请查看后确认是否可以开始编码。"
3. **等待用户明确确认**后才能进入 Stage 4
4. 如果用户有修改意见 → 返回 Stage 2 修改对应文档

**禁止行为**:
- ❌ 不得跳过确认直接开始编码
- ❌ 不得假设用户已确认
- ❌ 不得将"嗯"、"好"之外的模糊回复视为确认

**确认话术**:
```
📋 三份核心文档已完成：
  1. GDD (游戏设计文档) — docs/gdd.md
  2. 架构文档 — docs/architecture.md
  3. UI/UX 文档 — docs/uiux.md

请查看后回复"确认"开始编码，或提出修改意见。
⛔ 未确认前不会开始编码。
```

---

### Stage 4: TASK PLAN — 任务拆解

**目标**: 将三文档拆解为可执行的编码任务列表。

**准入**: Stage 3 用户已确认

**执行内容**:

1. 从架构文档提取模块列表
2. 按依赖关系排序（基础模块优先）
3. 估算每个模块的复杂度（低/中/高）
4. 生成任务清单

**任务清单格式**:
```markdown
# 任务计划 — [游戏名]

## 实现顺序

| 序号 | 模块 | 复杂度 | 依赖 | 状态 |
|------|------|--------|------|------|
| 1 | 场景搭建 | 低 | 无 | ⬜ |
| 2 | 角色控制 | 中 | 1 | ⬜ |
| 3 | 核心玩法 | 高 | 1,2 | ⬜ |
| 4 | UI 界面 | 中 | 1 | ⬜ |
| 5 | 音效系统 | 低 | 1 | ⬜ |
| 6 | 整合调试 | 中 | 全部 | ⬜ |
```

**准出**: 任务清单完成，依赖关系明确

---

### Stage 5: IMPLEMENT — 分模块实现

**目标**: 按任务计划逐个模块编码实现。

**准入**: Stage 4 任务清单完成

**执行规则**:

1. **严格按顺序实现**——不跳过依赖模块
2. **每完成一个模块必须调用 `build` 工具**（Rule #1）
3. **每个模块实现前**先重读三文档中相关部分
4. **遵循所有 UrhoX 引擎规则**（Rule #0 ~ Rule #14）
5. 更新任务清单状态（⬜ → 🔵 → ✅）

**模块实现检查清单**:
- [ ] 阅读了 GDD 中该模块的需求描述
- [ ] 阅读了架构文档中的模块设计
- [ ] 代码放在 `scripts/` 目录（Rule #1）
- [ ] 使用了正确的脚手架（Rule #2）
- [ ] 调用了 UrhoX MCP build 工具
- [ ] 没有引入 Rule 违规

---

### Stage 6: PREVIEW ⛔ — 预览确认门禁

**目标**: 强制暂停，让用户预览游戏效果。

**这是第二个强制暂停点**。AI 必须：

1. 确保最新代码已构建（调用 build 工具）
2. 提示用户在预览窗口查看效果
3. 明确询问："请在预览窗口查看游戏效果，确认是否满意。"
4. **等待用户明确确认**

**用户反馈处理**:

| 用户反馈 | 处理方式 |
|---------|---------|
| "满意" / "确认" | 进入 Stage 7 |
| "UI 需要调整" | 返回 Stage 5，修改 UI 模块 → 重新构建 → 回到 Stage 6 |
| "玩法需要修改" | 返回 Stage 2，更新 GDD → 重走 Stage 3~6 |
| "有 bug" | 在 Stage 5 修复 → 重新构建 → 回到 Stage 6 |

---

### Stage 7: QUALITY — 质量检查

**目标**: 对项目进行全面质量评估，必须达标才能交付。

**准入**: Stage 6 用户确认预览效果

**评分维度** (每项 0-20 分，满分 100 分):

| 维度 | 评分标准 | 权重 |
|------|---------|------|
| **引擎合规** | 遵循 Rule #0~#14，无违规 | 20 分 |
| **代码质量** | 命名规范、模块化、无冗余、类型标注 | 20 分 |
| **功能完整** | GDD 中所有功能均已实现 | 20 分 |
| **视觉体验** | UI 美观、动画流畅、视觉一致性 | 20 分 |
| **健壮性** | 无崩溃、边界处理、错误恢复 | 20 分 |

**评分规则**:
- ≥ 80 分: ✅ 通过，可进入 Stage 8
- 60-79 分: ⚠️ 需修复标记为"必修"的问题后重新评分
- < 60 分: ❌ 严重不达标，需返回 Stage 5 大幅修改

**评分报告格式**:

## 3. 专家角色系统

在流水线任何阶段，可切换为专家视角审视当前工作。

### 可用专家角色

| 角色 | 专长 | 何时召唤 |
|------|------|---------|
| **🎮 玩法设计师** | 核心循环、难度曲线、玩家体验 | Stage 2 设计 GDD 时 |
| **🏗️ 架构师** | 模块划分、代码结构、性能瓶颈 | Stage 2 设计架构时 |
| **🎨 UI/UX 设计师** | 界面布局、交互流程、视觉风格 | Stage 2 设计 UI 时 |
| **🔍 质量工程师** | 代码规范、引擎合规、边界测试 | Stage 7 质量检查时 |
| **⚡ 性能工程师** | 渲染优化、物理优化、内存管理 | Stage 5/7 性能问题时 |
| **📦 发布工程师** | 构建配置、发布材料、平台适配 | Stage 8 交付时 |

### 专家咨询流程

1. 在当前阶段遇到需要专业判断的问题时触发
2. 切换为对应专家视角
3. 以该专家的专业立场给出建议
4. 建议包含：结论 + 理由 + 具体操作步骤
5. 回到正常流水线继续执行

**专家咨询格式**:
```
🏗️ [架构师视角]

问题: 战斗系统应该用状态机还是行为树？

分析:
- 状态机: 适合状态数量 < 10 的简单系统，实现快
- 行为树: 适合复杂 AI，灵活但实现成本高

建议: 使用状态机
理由: 当前游戏的战斗状态只有 idle/attack/defend/die 四个，
      状态机完全够用，且 UrhoX 有 FSM 库支持。

操作步骤:
1. 参考 engine-docs/recipes/state-machine.md
2. 使用 /setup-fsm skill 配置动画状态机
3. 在 scripts/game/combat.lua 中实现状态逻辑
```

## 4. 返工机制

流水线不是单向的，支持有序返工。

### 返工触发条件与路径

| 触发场景 | 返工起点 | 返工路径 |
|---------|---------|---------|
| Stage 3: 用户不认可文档 | Stage 2 | 修改文档 → Stage 3 重新确认 |
| Stage 6: UI 效果不满意 | Stage 5 | 修改 UI 代码 → 构建 → Stage 6 重新预览 |
| Stage 6: 玩法需要大改 | Stage 2 | 更新 GDD → Stage 3 确认 → Stage 4~6 重走 |
| Stage 7: 质量不达标 | Stage 5 | 修复问题 → 构建 → Stage 7 重新评分 |

### 返工规则

1. **返工必须记录原因**——在对应文档中标注"变更原因"
2. **返工范围最小化**——只修改必要部分，不重做已通过的模块
3. **返工后必须重新通过门禁**——不跳过确认步骤
4. **返工次数上限**: 同一阶段最多返工 3 次，超过则建议重新评估需求

## 7. UrhoX 引擎规则集成

在 Stage 5 (IMPLEMENT) 和 Stage 7 (QUALITY) 中，必须检查以下引擎规则：

| 规则 | 检查内容 | 阶段 |
|------|---------|------|
| Rule #0 | 长度单位使用米 | Stage 5 |
| Rule #0.5 | Y-up 左手坐标系 | Stage 5 |
| Rule #1 | 代码放在 scripts/ 目录 | Stage 5 |
| Rule #2 | 基于脚手架开始 | Stage 5 |
| Rule #6 | NanoVG 使用 NanoVGRender 事件 | Stage 5/7 |
