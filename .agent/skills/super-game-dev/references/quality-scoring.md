# 质量评分细则 — Super Game Dev

Stage 7 (QUALITY) 质量检查的详细评分标准。

---

## 评分维度详解

### 1. 引擎合规 (20 分)

| 检查项 | 分值 | 扣分标准 |
|--------|------|---------|
| Rule #1: 代码在 scripts/ | 4 | 任何代码不在 scripts/ → -4 |
| Rule #2: 基于脚手架 | 2 | 从零开始未用脚手架 → -2 |
| Rule #6: NanoVG 事件正确 | 3 | 未用 NanoVGRender → -3 |
| Rule #7: NanoVG 字体创建 | 2 | 未创建字体直接绘文本 → -2 |
| Rule #9.6: 材质 Technique | 2 | 猜测 Technique 路径 → -2 |
| Rule #10: UI 系统选择 | 3 | 使用原生 UI 而非 urhox-libs/UI → -3 |
| Rule #12: 枚举值使用 | 2 | 使用数字代替枚举 → -2 |
| Rule #0.8: 分辨率处理 | 2 | 调用已禁用的分辨率设置 API → -2 |

### 2. 代码质量 (20 分)

| 检查项 | 分值 | 扣分标准 |
|--------|------|---------|
| 命名规范 | 4 | 变量名无意义(a,b,tmp) → 每处 -1，最多 -4 |
| 模块化程度 | 4 | 单文件超 1500 行未拆分 → -4 |
| 类型标注 | 4 | 空值声明无 @type 标注(Rule #11) → 每处 -1 |
| 代码重复 | 4 | 明显重复代码块(>10行) → 每处 -2 |
| 注释与可读性 | 4 | 复杂逻辑无注释 → 每处 -1 |

### 3. 功能完整 (20 分)

| 检查项 | 分值 | 评分方法 |
|--------|------|---------|
| GDD 功能覆盖率 | 12 | 已实现功能数/GDD 总功能数 × 12 |
| 核心循环完整 | 4 | 开始→游戏→结束→重新开始 完整 → 4 |
| 交互可用性 | 4 | 所有 UI 按钮/控件均可操作 → 4 |

### 4. 视觉体验 (20 分)

| 检查项 | 分值 | 扣分标准 |
|--------|------|---------|
| 配色一致性 | 5 | 颜色杂乱无主题 → -5 |
| 布局合理性 | 5 | UI 重叠/遮挡/溢出 → 每处 -2 |
| 动画/反馈 | 5 | 无任何交互反馈(按钮无响应态等) → -5 |
| 分辨率适配 | 5 | 不同屏幕比例下 UI 错位 → -5 |

### 5. 健壮性 (20 分)

| 检查项 | 分值 | 扣分标准 |
|--------|------|---------|
| 无崩溃 | 8 | 正常操作下崩溃 → -8 |
| 边界处理 | 4 | 数组越界/nil 访问未处理 → 每处 -2 |
| 状态管理 | 4 | 游戏状态混乱(暂停后仍可操作等) → -4 |
| 资源管理 | 4 | 事件未清理/内存泄漏 → 每处 -2 |

---

## 评分等级

| 分数区间 | 等级 | 处理方式 |
|---------|------|---------|
| 90-100 | ⭐ 优秀 | 直接交付 |
| 80-89 | ✅ 达标 | 可交付，建议改进 |
| 60-79 | ⚠️ 待改进 | 修复"必修"问题后重新评分 |
| 40-59 | ❌ 不达标 | 需要大幅修改 |
| 0-39 | 🚫 严重不达标 | 建议重新设计 |

---

## 必修 vs 建议

- **必修 (MUST FIX)**: 扣分 ≥ 3 的单项，或涉及崩溃/引擎违规
- **建议 (SHOULD FIX)**: 扣分 < 3 的单项，可在后续迭代中改进
