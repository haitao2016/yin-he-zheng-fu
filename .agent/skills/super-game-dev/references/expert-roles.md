# 专家角色指南 — Super Game Dev

在流水线的任何阶段，可切换为专家视角审视当前工作。
本文档定义每个专家角色的职责、关注点和输出格式。

---

## 角色定义

### 🎮 玩法设计师 (Gameplay Designer)

**关注点**:
- 核心循环是否有趣且可持续
- 难度曲线是否合理（不会太简单或太难）
- 玩家反馈是否即时且清晰
- 奖励机制是否有效激励玩家

**典型问题**:
- "这个核心循环玩家会觉得无聊吗？"
- "难度应该怎么递进？"
- "缺少什么反馈让玩家知道发生了什么？"

**输出模板**:
```
🎮 [玩法设计师视角]

评估: [当前玩法的整体评价]

优势:
- [列出做得好的方面]

改进建议:
1. [具体建议 + 理由]
2. [具体建议 + 理由]

参考: [同类游戏的做法参考]
```

---

### 🏗️ 架构师 (Architect)

**关注点**:
- 模块划分是否合理，职责是否清晰
- 数据流是否简洁，避免循环依赖
- 代码复用与扩展性
- UrhoX API 选型是否最优

**典型问题**:
- "这个模块划分是否合理？"
- "应该用状态机还是行为树？"
- "数据流有没有更简洁的设计？"

**UrhoX 特定关注**:
- 脚手架选择是否匹配游戏类型（Rule #2）
- 模块化拆分是否遵循 Rule #13（1500 行上限）
- 第三人称相机是否使用库（Rule #13.5）
- 多人/单机模式判断是否正确（Rule #14）

**输出模板**:
```
🏗️ [架构师视角]

问题: [待评估的架构决策]

选项分析:
- 方案 A: [描述] — 优点/缺点
- 方案 B: [描述] — 优点/缺点

建议: [推荐方案]
理由: [具体原因]

操作步骤:
1. [步骤 1]
2. [步骤 2]

参考文档: [相关 engine-docs/ 路径]
```

### 🎨 UI/UX 设计师 (UI/UX Designer)

**关注点**:
- 界面布局是否直观，控件是否易于操作
- 视觉风格是否一致
- 交互流程是否流畅
- 响应式适配是否完善

**UrhoX 特定关注**:
- 是否使用 urhox-libs/UI 组件（Rule #10）
- 分辨率适配方案（Rule #0.8）
- Flexbox 布局是否正确（flexShrink 陷阱）
- 是否参考了 UI 组件库（examples/14-ui-widgets-gallery.lua）

**输出模板**:
```
🎨 [UI/UX 设计师视角]

界面评估: [当前界面的整体评价]

布局建议:
- [具体的布局调整建议]

配色建议:
- 主色: [色值 + 用途]
- 辅色: [色值 + 用途]
- 强调色: [色值 + 用途]

交互改进:
1. [具体建议]
2. [具体建议]

参考组件: [建议使用的 UI 组件]
```

### 🔍 质量工程师 (QA Engineer)

**关注点**:
- 代码是否遵循引擎规则（Rule #0~#14）
- 边界条件是否处理
- 状态管理是否安全
- 错误恢复机制

**检查清单**:
- [ ] 事件订阅是否有对应的清理
- [ ] 数组操作是否从索引 1 开始（Rule #4）
- [ ] nil 值变量是否有类型标注（Rule #11）
- [ ] 枚举值是否使用常量（Rule #12）
- [ ] 游戏循环：开始→游玩→结束→重来 是否完整

**输出模板**:
```
🔍 [质量工程师视角]

扫描结果: [通过/需修复]

问题清单:
| 级别 | 文件 | 行号 | 问题描述 | 修复建议 |
|------|------|------|---------|---------|
| P0 | ... | ... | ... | ... |
| P1 | ... | ... | ... | ... |

合规检查:
- [✅/❌] Rule #X: [检查结果]
```

---

### ⚡ 性能工程师 (Performance Engineer)

**关注点**:
- 每帧操作的时间复杂度
- 对象创建与销毁频率
- 物理体数量与碰撞检测开销
- 渲染批次和 Draw Call

**UrhoX 特定关注**:
- NanoVG nvgCreateFont 是否只调用一次（Rule #7）
- 是否在 Update 中频繁创建临时对象
- 物理体数量是否在合理范围
- 纹理和模型资源是否复用

**输出模板**:
```
⚡ [性能工程师视角]

性能评估: [当前性能状况]

热点:
1. [性能热点描述 + 影响程度]
2. [性能热点描述 + 影响程度]

优化建议:
1. [具体优化步骤 + 预期效果]
2. [具体优化步骤 + 预期效果]
```

---

### 📦 发布工程师 (Release Engineer)

**关注点**:
- 构建配置是否正确
- 入口文件是否配置
- 资源引用是否完整
- 发布材料是否准备

**检查清单**:
- [ ] `scripts/main.lua` 包含 `Start()` 函数
- [ ] 所有资源在 `assets/` 目录且路径正确
- [ ] 无硬编码的调试代码
- [ ] 游戏名称和描述已确定
- [ ] 屏幕方向配置正确

**输出模板**:
```
📦 [发布工程师视角]

发布就绪度: [就绪/未就绪]

检查结果:
- [✅/❌] 构建配置
- [✅/❌] 入口文件
- [✅/❌] 资源完整性
- [✅/❌] 发布材料

待完成事项:
1. [具体事项]
```

## 专家召唤规则

1. **按需召唤**: 不必每个阶段都召唤所有专家
2. **阶段关联**: 优先召唤与当前阶段相关的专家
3. **冲突处理**: 不同专家意见冲突时，以玩法设计师为准（玩家体验优先）
4. **简明输出**: 专家建议控制在 200 字以内，给出可操作的具体步骤
