## 游戏画像扫描规范

### 扫描目录规则

```
扫描:
  ✅ scripts/          — 用户游戏代码
  ✅ docs/             — 已有文档（增量更新时参考）
  ✅ .project/settings.json — 多人模式判断

忽略:
  ❌ engine-docs/      — 引擎文档
  ❌ urhox-libs/       — 引擎工具库
  ❌ examples/         — 示例代码
  ❌ templates/        — 脚手架
  ❌ .emmylua/         — 类型定义
  ❌ dist/             — 发布产物
```

### 信息提取策略

| 画像字段 | 提取方法 |
|---------|---------|
| `genre` | 检查脚手架引用（scaffold-2d→2D休闲，scaffold-2d-physics→平台跳跃，scaffold-3d-character→3D角色）；检查物理组件（RigidBody2D→2D物理，RigidBody→3D物理）；检查相机类型 |
| `coreMechanic` | 分析 HandleUpdate/Update 中的核心循环；检查输入处理模式（触摸→触控操作，WASD→移动控制，鼠标→瞄准射击） |
| `orientation` | 检查 UI 布局方向；检查相机 aspect ratio 设置；检查 project.json 中的 screen_orientation |
| `techStack` | 收集所有 `require` 引用和 `CreateComponent` 调用 |
| `systems` | 列举 `scripts/` 下的所有模块文件，按目录结构推断子系统 |
| `artStyle` | 检查材质 Technique 类型（PBR→写实，NoTexture→程序化，NanoVG→矢量/像素）|
| `multiplayer` | 读取 `.project/settings.json` 的 `@runtime.multiplayer.enabled` |
| `balanceHints` | 提取 CONFIG 表中的数值常量（速度、生命值、伤害等） |

### 名称推断优先级

1. `project.json` 的 `taptap_publish.title`
2. `docs/gdd.md` 中的标题
3. 主入口文件名（去掉 main/game 前缀）
4. 询问用户

## 产物 A: 游戏设计文档

### 模板

```markdown
# {title} — 游戏设计文档

## 1. 概述
- **一句话描述**: {基于 coreMechanic 生成}
- **游戏类型**: {genre}
- **屏幕方向**: {orientation}
- **核心体验关键词**: {3-5 个词}

## 2. 核心玩法
### 游戏循环
{输入} → {核心动作} → {反馈} → {目标推进}

### 操作方式
| 输入 | 动作 |
|------|------|
| {从代码提取的输入映射} | {对应动作} |

### 胜负条件
- **胜利**: {从代码逻辑推断}
- **失败**: {从代码逻辑推断}

## 3. 游戏系统
| 系统 | 职责 | 关键模块 |
|------|------|---------|
| {systems[1]} | {从代码注释/函数名推断} | {文件名} |
| ... | ... | ... |

## 4. 技术方案
- **脚手架**: {scaffold 类型}
- **技术栈**: {techStack 列表}
- **代码规模**: {files} 个文件, {totalLines} 行

## 5. 数值参考
| 参数 | 当前值 | 来源 |
|------|--------|------|
| {balanceHints 中的参数} | {值} | {CONFIG 位置} |
```

### 生成规则

- 未从代码中推断出的字段标记为 `[待补充]`
- 系统关系图用文字描述，不生成 ASCII 图
- 里程碑章节不包含时间估算

## 产物 B: 商店描述

### 模板

```markdown
# {title} — TapTap 商店描述

## 文案

{一句话 Hook — 吸引注意力}

【核心玩法】
{2-3 句话描述核心玩法循环，玩家视角}

【游戏特色】
★ {特色1} — {一句话描述}
★ {特色2} — {一句话描述}
★ {特色3} — {一句话描述}

【适合谁玩】
{目标玩家群体，1-2 句话}

## taptap_publish 配置

```json
{
  "taptap_publish": {
    "title": "{title}",
    "description": "{上方文案的纯文本版}",
    "category": "{根据 genre 映射}",
    "screen_orientation": "{orientation}"
  }
}
```
```

### 分类映射

| genre | category |
|-------|----------|
| platformer | arcade |
| shooter | action |
| rpg | rpg |
| puzzle | puzzle |
| endless-runner | casual |
| card | card |
| strategy | strategy |
| simulation | simulation |
| racing | racing |
| fighting | action |
| other | casual |

### 写作规范

1. **玩家视角**: 描述体验而非技术实现
2. **具体生动**: "一刀斩落三个水果" 优于 "高伤害攻击"
3. **简洁有力**: 每句不超过 20 字
4. **动词优先**: "冲刺、跳跃、斩杀" 优于 "拥有冲刺功能"
5. **无 emoji**: 除非用户明确要求

## 产物 C: 开发日志

### 模板

```markdown
## 开发日志 — {YYYY-MM-DD}

### 本次摘要
{一句话总结核心变更}

### 变更详情
- [新增] {新功能描述}
- [修复] {修复内容}
- [优化] {优化内容}
- [调整] {数值/配置变更}
- [重构] {代码结构调整}
- [美术] {视觉相关更新}

### 技术备注
{架构决策或技术说明（如有）}

---
```

### 分类标签

| 标签 | 含义 | 触发条件 |
|------|------|---------|
| `[新增]` | 新功能 | 新增文件或新增函数 |
| `[修复]` | Bug 修复 | 修改了已有逻辑的错误 |
| `[优化]` | 性能/体验优化 | 性能改进、UX 改善 |
| `[调整]` | 数值/配置变更 | CONFIG 值变化 |
| `[重构]` | 代码结构调整 | 文件拆分、模块重组 |
| `[美术]` | 视觉相关 | 材质、贴图、UI 样式变更 |

### 增量规则

已有 `docs/devlog.md` 时：
- 新条目追加到文件**顶部**（最新在前）
- 保留已有条目不修改
- 日期使用当天日期

## 产物 D: 推广文案集

### 模板

```markdown
# {title} — 推广文案集

## 游戏推介（系列文案）

**推文 1/3 — 核心玩法**
{动词开头，描述核心动作}。{一句话说明玩法循环}。
{设问或反转，制造好奇心}

**推文 2/3 — 差异化卖点**
{与同类游戏的不同之处}。{具体的游戏场景描述}。
{情感共鸣句}

**推文 3/3 — 号召行动**
{总结核心体验}。{制造期待感}。
{号召语}

## 版本更新公告

### {版本号} 更新内容
{亮点功能 1}
{亮点功能 2}
{号召语}

## 特色功能介绍

### {功能名称}
{一句话核心价值}。
在 {场景} 中，你可以 {操作}。{反馈}，带来 {体验}。
```

### 写作原则

1. 每句不超过 20 字
2. 动词优先
3. 不承诺时间（不写"即将上线"）
4. 无 emoji（除非用户要求）
5. 三条推文聚焦不同维度（玩法/卖点/情感）

## 产物 E: 代码审查报告

### 模板

```markdown
# {title} — 代码审查报告

## 评分总览

| 类别 | 得分 | 说明 |
|------|------|------|
| 正确性 | ?/5 | API 使用是否正确 |
| 安全性 | ?/5 | 无崩溃/nil 风险 |
| 架构   | ?/5 | 模块化、职责分离 |
| 性能   | ?/5 | 无明显性能瓶颈 |
| 规范性 | ?/5 | 遵循 UrhoX 最佳实践 |

**综合评级**: {A/B/C/D}（A=优秀 B=良好 C=合格 D=需改进）

## P0 致命问题（共 N 项）

### [{文件}:{行号}] {问题标题}
**问题**: {描述}
**修复**: {建议}

## P1 重要问题（共 N 项）
...

## P2 建议优化（共 N 项）
...

## 推荐行动
1. {优先修复项}
2. ...
```

### 审查清单（精简版）

**P0 致命**:
- 数组索引从 0 开始
- eventData 访问方式错误
- NanoVG 不在 NanoVGRender 事件中
- nvgCreateFont 在渲染循环中
- 鼠标按钮用数字判断
- Box2D 碰撞体在子节点
- nil 变量成员访问

**P1 重要**:
- 调用已禁用的分辨率 API
- 资源路径加了目录前缀
- 硬编码数字枚举
- 手动计算第三人称相机
- 使用原生 UI 系统
- 单文件超 1500 行

**P2 建议**:
- table.unpack 位置陷阱
- 全局变量污染
- 重复代码
- 魔法数字
- 缺少 Cleanup

### 综合评级算法

```
P0 > 0        → D
P0 = 0, P1 > 3 → C
P0 = 0, P1 ≤ 3 → B
P0 = 0, P1 = 0 → A
```

## 一致性校验规则

批量生成完成后，执行以下一致性校验：

| 校验项 | 规则 |
|--------|------|
| 游戏名称 | 所有产物中 `{title}` 相同 |
| 游戏类型 | GDD 中的类型 = 商店描述的 category 映射 |
| 屏幕方向 | GDD = 商店描述 = 技术摘要 |
| 核心玩法描述 | GDD 的核心玩法 ≈ 商店描述的核心玩法（措辞不同但语义一致）|
| 系统列表 | GDD 中的系统 ⊇ 技术摘要中的模块 |
| 多人模式 | 所有产物中一致 |

校验失败时，以第一阶段的游戏画像为基准修正。
