# Game Dev Planner — UrhoX Lua 游戏开发规划师

基于 [lobster-dev-planner](https://clawhub.ai/wangxiaofei860208-source/lobster-dev-planner) 的 AI 驱动开发规划系统，
适配为 UrhoX Lua 游戏开发的全流程引导式规划与执行框架。

通过对话式需求收集 → 游戏设计文档生成 → 分阶段开发执行 → 测试验收交付的完整流水线，
帮助用户从一个模糊的游戏想法走到可运行的完整游戏。

## Trigger

当用户表达以下意图时触发：
1. "帮我做一个游戏" / "我想做一个 XXX 游戏"
2. "帮我规划一个项目" / "plan 模式"
3. "从头开始开发" / "我有一个游戏想法"
4. "不知道从哪开始" / "帮我梳理需求"
5. 用户描述了一个游戏概念但没有具体的实现方向
6. 用户需要完整的游戏设计文档（GDD）

**与 game-creation-workflow 的关系**: game-creation-workflow 侧重代码实现层面的脚手架选择和代码编写流程；本 Skill 侧重**前置规划阶段**的需求引导、文档生成和开发执行管理，两者互补。

## Context

本 Skill 提供 7 阶段的游戏开发全流程管理：

```
┌─────────────────────────────────────────────────────────┐
│  Phase 1  PLAN MODE    对话收集需求（小白友好选项引导）   │
│  Phase 2  DESIGN DOC   生成游戏设计文档套件（GDD三件套） │
│  Phase 3  SCAFFOLD     选择脚手架 + 项目结构初始化       │
│  Phase 4  DEV PLAN     拆分开发节点 + 依赖排序          │
│  Phase 5  EXECUTE      逐节点开发 + 每节点自测          │
│  Phase 6  POLISH       打磨优化 + 完整测试              │
│  Phase 7  DELIVERY     构建发布 + 交付报告              │
└─────────────────────────────────────────────────────────┘
```

## §1 Phase 1：PLAN MODE — 对话式需求收集

### 对话原则

- 用选项卡降低用户决策负担（不懂技术也能选）
- 每一步最多展示 8 个选项，避免信息过载
- 用户已描述清楚的内容直接跳过对应步骤
- 全程用平实语言，不使用引擎术语

### Step 0：开场白

如果用户已描述需求，直接跳到 Step 1 解析。否则：

```
你好！我是你的游戏开发规划师。
告诉我你想做什么游戏，我会帮你：
  1. 规划整个游戏的设计方案
  2. 生成完整的游戏设计文档
  3. 分步骤把游戏做出来

先说说吧 — 你脑子里想做什么样的游戏？
（不需要懂技术，想到什么说什么就行）
```

### Step 1：游戏类型

```
根据你的描述，确认一下游戏类型：
A. 2D 休闲游戏 — Flappy Bird、贪吃蛇那种
B. 2D 平台跳跃 — 马里奥、Celeste 那种
C. 2D 射击游戏 — 弹幕、太空射击那种
D. 3D 动作游戏 — 第三人称砍杀、射击
E. 3D 场景展示 — 建筑漫游、产品展示
F. 棋牌/益智游戏 — 扫雷、2048、消消乐
G. RPG/冒险游戏 — 有故事、有战斗
H. 模拟经营 — 开店、种田、城市建造
I. 其他（告诉我）
```

| H | scaffold-2d.lua 或 scaffold-3d-scene.lua | 2D/3D 俯视 | 无/简单 |

### Step 2：游戏规模


## §4 Phase 4：DEV PLAN — 开发节点拆分

将 GAME_DESIGN.md §6 的节点规划转化为可执行的开发任务单：

### 4.1 节点拆分原则

1. **每节点产出可验证** — 每个节点完成后游戏必须可以 build 成功
2. **先核心后外围** — 核心玩法 → UI → 音效 → 特效 → 打磨
3. **依赖关系明确** — 无依赖的节点可并行
4. **单节点代码量 < 300 行** — 超过就拆分

### 4.2 标准节点序列（模板）

```
节点 1: 项目初始化 + 基础场景搭建
  依赖：无
  产出：空场景可运行，相机和灯光就位

节点 2: 核心玩法（第一部分）
  依赖：节点 1
  产出：玩家可操控，核心交互可用

节点 3: 核心玩法（第二部分）
  依赖：节点 2
  产出：完整的游戏循环（开始→游玩→结束）

节点 4: UI / HUD
  依赖：节点 3
  产出：分数显示、生命值、游戏状态提示

节点 5: 音效 + BGM
  依赖：节点 3
  产出：关键操作有音效反馈

节点 6: 关卡/难度/进度系统
  依赖：节点 3
  产出：难度递增、关卡切换或无尽模式

节点 7: 存档 / 排行榜（如需要）
  依赖：节点 3, 4
  产出：分数保存、排行榜展示

节点 8: 打磨优化
  依赖：全部
  产出：动画效果、屏幕震动、粒子特效

节点 9: 构建 + 测试
  依赖：全部
  产出：最终 build 通过，交付报告
```

### 4.3 节点任务单格式

```
┌────────────────────────────────────────────┐
│  节点 {X}/{N}：{节点名称}                    │
│  ─────────────────────────────────────     │
│  依赖节点：{列表或"无"}                     │
│                                            │
│  参考文档：                                 │
│    GAME_DESIGN.md → §{X} {章节}            │
│    TECH_SPEC.md → §{X} {模块接口}          │
│                                            │
│  交付要求：                                 │
│    * {具体功能点 1}                         │
│    * {具体功能点 2}                         │
│    * build 工具验证通过                     │
│                                            │
│  UrhoX 关键点：                            │
│    * {需要注意的引擎规则，如 Rule #X}        │
│    * {推荐使用的 API 或库}                  │
│                                            │
│  验收标准：                                 │
│    * {可观测的验证条件}                     │
└────────────────────────────────────────────┘
```

## §5 Phase 5：EXECUTE — 逐节点开发

### 5.1 执行流程

对每个节点执行以下步骤：

```
读取节点任务单
    ↓
查阅 UrhoX 引擎文档（engine-docs/、examples/）
    ↓
编写代码（遵守 CLAUDE.md 所有规则）
    ↓
调用 build 工具验证
    ↓
输出节点完成报告
    ↓
更新文档（如有变更）
    ↓
进入下一节点
```

### 5.2 节点完成报告格式

每个节点完成后输出：

```
═══════════════════════════════════════════════
  节点 {X}/{N}  [{节点名}]  完成
═══════════════════════════════════════════════
完成内容
───────────────────────────────────────────
  * {功能点 1} — {对应文件}
  * {功能点 2} — {对应文件}

文件变更
───────────────────────────────────────────
  新建：{文件路径}  ({说明})
  修改：{文件路径}  ({改动摘要})

构建结果
───────────────────────────────────────────
  build 工具：通过
  LSP 检查：0 错误，{N} 警告

自测结果
───────────────────────────────────────────
  * {验收条件 1} — 通过
  * {验收条件 2} — 通过

遵守的引擎规则
───────────────────────────────────────────
  * Rule #{N}: {规则名} — 已遵守
  * {其他注意事项}

→ 下一步：节点 {X+1}：{下一节点名}...
═══════════════════════════════════════════════
```

### 5.3 开发过程中的引擎规则检查清单

每个节点开发时必须对照：

| 检查项 | 规则 | 说明 |
|--------|------|------|
| 代码放在 scripts/ | Rule #1 | 不要写到其他目录 |
| 使用脚手架 | Rule #2 | 不从空白开始 |
| 数组从 1 开始 | Rule #4 | Lua 数组索引 |
| NanoVG 用 NanoVGRender 事件 | Rule #6 | 不用 Update 事件 |
| NanoVG 先创建字体 | Rule #7 | nvgCreateFont 只调用一次 |
| 使用枚举不用数字 | Rule #12 | MOUSEB_LEFT 而非 0 |
| UI 使用 urhox-libs/UI | Rule #10 | 不用原生 UI |
| 每次改动后 build | Rule #1 (系统) | 调用 build 工具 |
| eventData 用 GetInt/GetFloat | Rule #3 | tolua++ 绑定 |
| 材质用 PBRNoTexture | Rule #9.6 | 不猜路径 |

## §6 Phase 6：POLISH — 打磨优化

核心功能完成后进入打磨阶段：

### 6.1 打磨检查清单

```
【视觉打磨】
  □ 颜色是否统一且美观（不是默认灰色）
  □ UI 布局是否整齐、间距是否合理
  □ 字体大小层级是否清晰（标题 > 正文 > 注释）
  □ 有没有必要的动画过渡效果

【操作手感】
  □ 输入响应是否即时（无延迟感）
  □ 关键操作是否有音效反馈
  □ 碰撞/交互是否符合直觉
  □ 难度曲线是否平滑

【稳定性】
  □ 快速操作是否会崩溃
  □ 边界情况是否处理（空数组、nil 值）
  □ 长时间运行是否稳定

【平台适配】
  □ 手机触摸和 PC 键鼠是否都支持
  □ 横屏/竖屏是否正确
  □ 不同分辨率下 UI 是否正常
```

## §8 文档标准

### 文档详细度原则

**标准：任何 AI 助手拿到这三份文档，不问任何问题就能继续开发。**

| 维度 | 要求 |
|------|------|
| 模块接口 | 每个公开函数有参数类型、返回值、一句话说明 |
| 事件契约 | 发送者、监听者、数据字段全部明确 |
| 资源路径 | 每个资源有完整路径和加载方式 |
| 引擎规则 | 标注适用的 Rule 编号 |
| 验收条件 | 每个节点有可观测的通过条件 |

### 文档更新规则

- 每个节点完成后，如果实际实现与文档有偏差，**立即更新文档**
- 新增的模块/接口必须同步到 TECH_SPEC.md
- 新增的资源必须同步到 ASSET_LIST.md
- 文档是**活文档**，始终反映项目的最新状态

## §9 常见问题与排错

| 场景 | 处理方式 |
|------|---------|
| 用户需求模糊 | 通过 Step 1-7 的选项引导逐步明确 |
| 用户中途改需求 | 更新 GAME_DESIGN.md → 重新评估受影响的节点 → 从受影响节点继续 |
| 节点 build 失败 | 检查 CLAUDE.md 的规则检查清单 → 查阅引擎文档 → 修复后重新 build |
| 代码超过 1000 行 | 立即拆分模块（Rule #13）→ 更新 TECH_SPEC.md 文件结构 |
| 不确定用哪个 API | 查 examples/api-index.md → 查 engine-docs/api/ → 读相关示例 |
| 资源生成失败 | 调整提示词 → 重试 → 用占位资源先继续开发 |
| 用户说"太复杂了" | 砍掉非核心功能 → 先做 MVP → 后续迭代添加 |
| 多人模式需求 | 先读 .project/settings.json → 根据 multiplayer.enabled 决定架构 |
