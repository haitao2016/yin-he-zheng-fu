# Iterative Image Craft — AI 图像迭代生产系统

## 核心价值

| 痛点 | 本 Skill 解决方案 |
|------|------------------|
| 一次生成质量不达标，反复重试浪费时间 | **三阶段流水线**：DRAFT(1K) → ITERATE → FINALIZE(2K/4K) |
| 高分辨率生成慢且费资源 | **分辨率阶梯**：草稿用 1K 快速验证，定稿才上 2K/4K |
| Prompt 写不好导致效果差 | **Prompt 模板**：六维度生成模板 + 三种编辑模板 |
| 文件命名混乱，找不到历史版本 | **命名规范**：`{描述名}_{时间戳}.png`，阶段标记清晰 |
| 生成前缺少检查导致无效调用 | **前置检查**：类型/用途/比例/源图 四项预检 |

---

## 触发条件

### MUST trigger when

| 场景 | 说明 |
|------|------|
| 用户要求**反复迭代**图像 | "先出个草稿看看"、"调一下颜色"、"再改改" |
| 用户要求**编辑现有图像** | "把背景换成蓝色"、"去掉水印"、"改成卡通风格" |
| 用户要求**不同分辨率版本** | "先低分辨率看看效果"、"出高清版" |
| 用户对生成质量有**明确迭代意图** | "这个不太好，调整一下"、"差不多了，出最终版" |

### SKIP when

| 场景 | 路由到 |
|------|--------|
| 一次性批量生成大量素材（无迭代需求） | `auto-game-assets` |
| 从游戏概念出发生成完整素材集 | `game-cog` |
| 需要跨模态资产管线编排 | `ai-asset-pipeline` |
| 单张图像一次生成即可，无需迭代 | 直接调用 `generate_image` |
| 查找/下载免费素材 | `game-assets-finder` |

## 三阶段流水线

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  Phase 1    │     │  Phase 2    │     │  Phase 3    │
│  DRAFT      │────▶│  ITERATE    │────▶│  FINALIZE   │
│  1K 快速验证 │     │  调整优化    │     │  2K/4K 定稿  │
└─────────────┘     └─────────────┘     └─────────────┘
```

### Phase 1: DRAFT — 快速生成草稿

**目标**：用最低成本验证构图、风格、主体是否正确。

**执行步骤**：

1. **前置检查**（见下方 Preflight Checklist）
2. 使用 `generate_image` 工具，参数：
   - `resolution`: `"1K"` （草稿阶段固定 1K）
   - `aspect_ratio`: 根据素材类型选择（见分辨率策略表）
   - `target_size`: 根据最终用途选择（见分辨率策略表）
   - `prompt`: 使用生成 Prompt 模板构建
3. 将生成结果展示给用户，询问反馈
4. 记录当前 prompt 和参数，供迭代阶段使用

**完成标志**：用户确认"方向对了"或"大致可以"。

### Phase 2: ITERATE — 迭代优化

**目标**：根据用户反馈精细调整，直到满意。

**执行规则**：

| 修改范围 | 策略 | 工具 |
|---------|------|------|
| 微调（颜色、亮度、细节措辞） | 调整 prompt 关键词，保持其他参数不变 | `generate_image` |
| 局部修改（换背景、加元素、去水印） | 使用图像编辑 | `edit_image` |
| 大改（换风格、换主体、换构图） | 重写 prompt，重新生成 | `generate_image` |
| 多方案对比 | 同一 prompt 不同 seed，或多个 prompt 变体 | `generate_image`（多次）|

**迭代上限**：单张图像最多 **5 轮**迭代。超过 5 轮仍不满意时：
- 总结当前问题
- 建议用户提供参考图（`reference_images`）
- 或建议调整需求方向

**完成标志**：用户确认"可以了"、"就这样"、"出最终版"。

### Phase 3: FINALIZE — 高清定稿

**目标**：用确认的 prompt 和参数，生成高分辨率最终版。

**执行步骤**：

1. 使用 Phase 2 最终确认的 prompt（不做任何修改）
2. 提升 `resolution` 参数：
   - 游戏纹理/角色：`"2K"`
   - 宣传图/壁纸/高精度需求：`"4K"`
3. 可选：设置 `thinking_level: "high"` 提升复杂 prompt 的生成质量
4. 最终文件按命名规范保存

**完成标志**：高分辨率文件生成并展示给用户。

## 分辨率策略

### 按素材类型选择 aspect_ratio

| 素材类型 | 推荐 aspect_ratio | 推荐 target_size | 说明 |
|---------|-------------------|-----------------|------|
| 游戏图标 | `1:1` | `128x128` / `256x256` | 正方形图标 |
| 角色立绘 | `2:3` 或 `3:4` | `512x768` / `512x1024` | 竖版人物 |
| 场景背景（横版） | `16:9` 或 `3:2` | `1024x576` / `1024x768` | 横版场景 |
| 场景背景（竖版） | `9:16` 或 `3:4` | `576x1024` / `768x1024` | 竖版场景 |
| 纹理贴图 | `1:1` | `512x512` / `1024x1024` | 正方形纹理 |
| UI 元素 | `1:1` 或 `4:3` | `256x256` / `512x512` | 界面元素 |
| 宣传图 | `16:9` | `1920x1080` | 横版宣传 |

### 分辨率阶梯

| 阶段 | resolution 参数 | 用途 |
|------|----------------|------|
| DRAFT | `"1K"` | 快速验证构图和风格 |
| ITERATE | `"1K"` | 保持快速迭代（除非用户要求看高清细节） |
| FINALIZE | `"2K"` 或 `"4K"` | 最终交付 |

## Prompt 模板

### 生成 Prompt 模板（六维度）

构建 `generate_image` 的 `prompt` 参数时，按以下六个维度组织：

```
[主体] + [风格] + [构图] + [光照] + [背景] + [色调]
```

| 维度 | 说明 | 示例 |
|------|------|------|
| 主体 | 画面核心对象 | "一只橙色小猫"、"中世纪骑士" |
| 风格 | 艺术风格 | "像素风格"、"水彩风"、"赛博朋克" |
| 构图 | 视角和布局 | "正面特写"、"俯视全景"、"居中对称" |
| 光照 | 光线效果 | "柔和自然光"、"霓虹灯光"、"逆光剪影" |
| 背景 | 环境描述 | "透明背景"、"星空"、"森林" |
| 色调 | 整体色彩倾向 | "暖色调"、"冷蓝色调"、"高饱和" |

**示例**：
```
prompt: "一只橙色小猫，像素风格，正面居中，柔和自然光，透明背景，暖色调"
```

**注意**：prompt 必须是**简短中文描述**（max 50KB），不要写英文长句。

### 编辑 Prompt 模板

用于 `edit_image` 工具的 `prompt` 参数：

**模板 A — 局部修改**：
```
prompt: "把[目标区域]改成[新内容]"
示例: "把背景改成星空"、"把帽子改成红色"
```

**模板 B — 风格转换**：
```
prompt: "把图片风格改成[目标风格]"
示例: "把图片风格改成水彩画"、"改成像素风格"
```

**模板 C — 添加/移除元素**：
```
prompt: "添加[新元素]" 或 "去掉[目标元素]"
示例: "添加一个太阳"、"去掉水印"、"在角色手中添加一把剑"
```

## 前置检查清单（Preflight Checklist）

### 新图像生成前

在调用 `generate_image` 之前，确认以下信息：

| # | 检查项 | 确认方式 |
|---|--------|---------|
| 1 | 素材类型（图标/立绘/场景/纹理/UI） | 询问用户或从上下文推断 |
| 2 | 用途（游戏内使用/宣传/占位） | 决定最终 target_size |
| 3 | 是否需要透明背景 | prompt 含"透明"则设 `transparent: true` |
| 4 | aspect_ratio 选择 | 参照分辨率策略表 |
| 5 | 是否有参考图 | 有则使用 `reference_images` 参数 |

### 编辑图像前

在调用 `edit_image` 之前，确认以下信息：

| # | 检查项 | 确认方式 |
|---|--------|---------|
| 1 | 源图像路径是否存在 | 检查 `image` 参数指向的文件 |
| 2 | 修改意图是否明确 | 确认 prompt 描述清晰 |
| 3 | 输出尺寸是否需要变化 | 默认保持原图尺寸 |
| 4 | 是否需要参考图辅助 | 风格一致性需求时使用 `reference_images` |

## 文件命名规范

### 命名格式

工具自动生成的文件名格式为 `{name}_{timestamp}.png`（生成）或 `edited_{name}_{timestamp}.png`（编辑）。

**`name` 参数命名建议**：

| 阶段 | name 命名模式 | 示例 |
|------|-------------|------|
| DRAFT | `{描述}_草稿` | `金币图标_草稿` |
| ITERATE | `{描述}_v{轮次}` | `金币图标_v2`、`金币图标_v3` |
| FINALIZE | `{描述}_定稿` | `金币图标_定稿` |
| EDIT | `{描述}_编辑` | `角色立绘_编辑` |

### 文件组织结构

生成的素材按用途存放在 `assets/` 对应子目录：

```
assets/
├── Textures/      # 纹理贴图
├── Icons/         # 游戏图标
├── UI/            # UI 元素
├── Characters/    # 角色立绘
├── Backgrounds/   # 场景背景
└── Promo/         # 宣传素材
```

## 编辑策略指南

### 按场景选择工具

| 场景 | 推荐工具 | 说明 |
|------|---------|------|
| 换背景 | `edit_image` | prompt: "把背景改成..." |
| 换颜色/色调 | `edit_image` | prompt: "把颜色改成..." |
| 添加元素 | `edit_image` | prompt: "添加一个..." |
| 去除元素 | `edit_image` | prompt: "去掉..." |
| 换风格（保持内容） | `edit_image` | prompt: "改成...风格" |
| 完全重画 | `generate_image` | 重写 prompt，使用原图作 reference |

### reference_images 使用场景

| 场景 | 用法 |
|------|------|
| 风格一致性 | 将已确认风格的图作为 reference，生成同风格新图 |
| 角色一致性 | 将角色正面图作为 reference，生成不同姿势/场景 |
| 批量生成同系列 | 将第一张定稿作为 reference，批量生成其他素材 |
| 编辑时保持特征 | 将原图作为 reference，编辑后保持角色/物体特征 |

## 与其他 Skill 协作

| 场景 | 本 Skill 的角色 | 协作 Skill |
|------|----------------|-----------|
| 从概念到素材集 | 对 game-cog 生成的单张图做迭代打磨 | `game-cog`（概念生成） |
| 批量素材生产 | 打磨代表作后，auto-game-assets 批量复制风格 | `auto-game-assets`（批量生成） |
| 资产管线 | 作为 pipeline 中"图像打磨"环节 | `ai-asset-pipeline`（管线编排） |
| 像素风素材 | 先用本 skill 迭代确认风格，再交 pixel-art 批量 | `@game_pixel-art-generator`（像素画） |
| 游戏发布素材 | 迭代宣传图/图标后交 game-material 上传 | `generate_game_material`（发布素材） |
| 角色立绘集 | 迭代确认一张代表风格后，交 character-portraits 批量 | `@m-mikoto_character-portraits`（角色立绘集） |

**典型组合流程**：
```
game-cog（概念 → 素材方向）
    ↓
iterative-image-craft（单张迭代打磨 → 确认风格）
    ↓
auto-game-assets / batch_generate_images（批量生成同风格素材）
    ↓
ai-asset-pipeline（质量门控 → 交付）
```

## 实践示例

### 示例 1：游戏图标迭代

```
用户："帮我做一个金币图标"

Phase 1 - DRAFT:
  前置检查：类型=图标，用途=游戏内，无透明需求，1:1，无参考
  调用 generate_image:
    prompt: "金色游戏金币，像素风格，正面居中，金属光泽，透明背景，高饱和金色"
    aspect_ratio: "1:1"
    target_size: "256x256"
    resolution: "1K"
    transparent: true
    name: "金币图标_草稿"

用户："颜色太暗了，要更亮一些"

Phase 2 - ITERATE (v2):
  策略：微调 → 调整 prompt 关键词
  调用 generate_image:
    prompt: "明亮金色游戏金币，像素风格，正面居中，强烈金属高光，透明背景，明亮鲜艳金色"
    （其他参数不变）
    name: "金币图标_v2"

用户："可以了，出最终版"

Phase 3 - FINALIZE:
  调用 generate_image:
    （使用 v2 确认的 prompt，不做修改）
    resolution: "2K"
    name: "金币图标_定稿"
```

### 示例 2：图像编辑

```
用户："把这张角色图的背景换成森林"

前置检查：源图存在 ✓，意图明确 ✓，尺寸不变 ✓
调用 edit_image:
  image: "assets/Characters/warrior.png"
  prompt: "把背景换成茂密的奇幻森林，保持角色不变"
  target_size: "512x768"
  name: "战士_森林背景"

用户："森林太暗了"

ITERATE:
  调用 edit_image:
    image: （使用上一轮编辑结果的路径）
    prompt: "把森林背景调亮，增加阳光照射效果"
    name: "战士_森林背景_v2"
```
