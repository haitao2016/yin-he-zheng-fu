# UrhoX Lua 代码审查清单

> 结构化开发会话 Phase 6 (REVIEW) 使用的完整审查清单。
> 按类别逐项检查，标记问题等级。

---

## 使用方法

1. 在 Phase 6 中逐项检查以下清单
2. 每项标记状态：✅ 通过 / 🔴 阻断 / 🟡 建议 / ⬜ 不适用
3. 所有 🔴 项必须修复后才能进入 Phase 7
4. 🟡 项尽量修复，无法修复时记录到会话报告

---

## 一、正确性检查

### 1.1 功能完整性

- [ ] 所有用户要求的功能点均已实现
- [ ] 边界情况已处理（空值、极端输入、溢出等）
- [ ] 没有遗留的 TODO 或占位代码
- [ ] 游戏逻辑与用户描述一致

### 1.2 构建验证

- [ ] 已调用 UrhoX MCP `build` 工具
- [ ] 构建无错误
- [ ] 构建无严重警告（允许的警告除外）

### 1.3 运行时安全

- [ ] 无除零风险（除法操作前检查分母）
- [ ] 无数组越界风险（索引从 1 开始）
- [ ] 无 nil 解引用风险（可能为 nil 的值使用前检查）
- [ ] 无无限循环风险
- [ ] 事件订阅有对应的取消订阅（如果组件会被销毁）

---

## 二、引擎合规检查

### 2.1 核心规则

- [ ] 代码存放在 `scripts/` 目录
- [ ] 未修改 `urhox-libs/` 目录（只读）
- [ ] 未写入 `/workspace/dist/` 目录
- [ ] 未使用 `graphics:SetMode()`（已禁用）
- [ ] 未使用 `io` 库（已被沙箱移除，使用 `File` 替代）
- [ ] 未提及 "Urho3D"（使用 "UrhoX"）

### 2.2 坐标系与单位

- [ ] 长度单位使用米（重力 -9.81，角色高度 1.5-2.0）
- [ ] Y-up 左手坐标系（Y=上，Z=前，X=右）
- [ ] 旋转使用正确的轴（Yaw=Y 轴，Pitch=X 轴）

### 2.3 Lua 语言规范

- [ ] 数组索引从 1 开始（非 0）
- [ ] 循环使用 `for i = 1, n do`（非 `for i = 0, n-1 do`）
- [ ] `table.unpack()` 只在表构造器最后位置使用
- [ ] `eventData` 使用正确的访问方式：
  - `eventData["Key"]:GetInt()` 或 `eventData:GetInt("Key")`

### 2.4 资源路径

- [ ] 资源路径不带 `assets/` 前缀
  - ✅ `cache:GetResource("Texture2D", "Textures/player.png")`
  - ❌ `cache:GetResource("Texture2D", "assets/Textures/player.png")`
- [ ] 模块引用使用正确格式
  - ✅ `require "urhox-libs.Platform.PlatformUtils"`
  - ✅ `require "Utils.Helper"`（scripts/ 下的模块）

### 2.5 NanoVG

- [ ] NanoVG 渲染在 `NanoVGRender` 事件中执行（非 Update）
- [ ] 字体在初始化时创建一次（`nvgCreateFont` 不在每帧调用）
- [ ] `nvgBeginFrame` / `nvgEndFrame` 配对使用
- [ ] 文本绘制前设置了字体（`nvgFontFace` + `nvgFontSize`）

### 2.6 UI 系统

- [ ] 使用 `urhox-libs/UI` 而非原生 Urho3D UIElement
- [ ] UI 不与 raw NanoVG 混用（避免层级冲突）
- [ ] `UI.Init()` 包含字体配置
- [ ] `UI.Scale.DEFAULT` 用于默认缩放策略

### 2.7 物理系统

- [ ] Box2D 碰撞形状在同一刚体节点上（非子节点）
- [ ] 使用枚举常量（`BT_STATIC`, `BT_DYNAMIC`）而非数字
- [ ] 3D 物理碰撞事件订阅正确

### 2.8 输入处理

- [ ] 鼠标按钮使用枚举（`MOUSEB_LEFT`，非 `0`）
- [ ] 键盘按键使用枚举（`KEY_SPACE`，非数字）
- [ ] FPS/TPS 游戏设置了 `input.mouseMode = MM_RELATIVE`

### 2.9 3D 模型

- [ ] 模型尺寸通过 `boundingBox.size` 获取或查阅文档（非猜测）
- [ ] 缺失的基础形状使用 `CustomGeometry`
- [ ] 程序化材质使用 `PBRNoTexture` 系列 Technique

### 2.10 相机

- [ ] 第三人称相机使用 `ThirdPersonCamera` 库
- [ ] 正交相机 `orthoSize` 计算考虑 0.5 因子

## 三、游戏质量检查

### 3.1 用户体验

- [ ] 游戏有基本的视觉反馈（按钮点击、碰撞效果等）
- [ ] 文本清晰可读（字号合适、颜色对比度足够）
- [ ] 无明显的卡顿或延迟
- [ ] 游戏有明确的胜利/失败条件（如适用）

### 3.2 错误处理

- [ ] 异常状态有合理的降级处理
- [ ] 不会因为用户操作导致程序崩溃
- [ ] 关键操作有日志输出（首次交付时）

### 3.3 性能

- [ ] 没有在 Update 循环中创建资源（Material、Model 等）
- [ ] 没有在 Update 循环中创建临时表/闭包（频繁 GC）
- [ ] 大量实体使用对象池模式
- [ ] NanoVG 字体创建不在每帧循环中

---

## 四、代码质量检查

### 4.1 可读性

- [ ] 变量命名有意义（非 `a`, `b`, `temp`）
- [ ] 函数职责单一，不超过 50 行（建议）
- [ ] 复杂逻辑有必要的注释
- [ ] 文件结构清晰（初始化 → 逻辑 → 事件处理）

### 4.2 类型标注

- [ ] 未赋值或赋 nil 的变量有 `---@type` 标注
- [ ] 事件处理函数参数有类型标注
- [ ] 关键函数有 `---@param` 和 `---@return` 标注

### 4.3 模块化

- [ ] 单文件不超过 1500 行（超过则必须拆分）
- [ ] 模块间通过 `require` 引用（非全局变量传递）
- [ ] 全局变量使用最小化

### 4.4 代码风格

- [ ] 缩进一致（推荐 4 空格或 2 空格，不混用）
- [ ] 没有被注释掉的死代码（已删除则彻底删除）
- [ ] 没有无用的 `require`
- [ ] 常量提取到文件顶部或配置模块

---

## 五、多人游戏特定检查

> 仅当 `.project/settings.json` 中 `multiplayer.enabled = true` 时执行

### 5.1 架构

- [ ] 客户端代码在 `Client.lua`，服务端代码在 `Server.lua`
- [ ] 游戏状态由服务端管理
- [ ] 客户端不信任本地数据

### 5.2 网络安全

- [ ] 关键计算在服务端执行
- [ ] 客户端发送的数据有校验
- [ ] 无明显的作弊漏洞
