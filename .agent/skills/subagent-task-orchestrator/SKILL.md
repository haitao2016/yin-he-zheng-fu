# Subagent Task Orchestrator — 子代理任务编排器

> 灵感源自 [superpowers-subagent-dev](https://clawhub.ai/axelhu/superpowers-subagent-dev) 的
> subagent dispatch 模式，适配为 UrhoX 沙箱环境下的多任务编排系统。
>
> 核心理念：**每个子任务 dispatch 独立子代理 + 两阶段审查（合规 → 质量）**
>
> - 任务 dispatch 模板 → `references/dispatch-templates.md`
> - 两阶段审查清单 → `references/review-checklist.md`

---

## 与现有 Skill 的关系

| Skill | 定位 | 本 Skill 的差异 |
|-------|------|----------------|
| `structured-dev-session` | 单任务 7 阶段门控流水线 | 本 Skill 编排**多个任务**，每个任务由子代理执行 |
| `game-dev-planner` | 需求收集 → GDD → 分步开发 | 本 Skill 在计划完成后接管**执行层**的子代理调度 |
| `auto-workflow` | 重复模式检测 + 代码生成 | 本 Skill 侧重**任务级编排**，而非代码模式自动化 |
| `dev-tools-pack` | 6 个独立生产力工具 | 本 Skill 是**流程编排器**，可调用这些工具 |
| `@vintlin_team-dev` | 六阶段流水线多任务管理 | 本 Skill 增加了**子代理 dispatch**和**两阶段审查** |

**互补关系**：
- `game-dev-planner` 生成计划 → **本 Skill** 执行计划中的每个任务
- `dev-tools-pack` 的 Code Review 工具可作为本 Skill 审查阶段的参考清单

## 核心架构

```
+---------------------------------------------------------------------+
|                      主代理 (Controller)                             |
|                                                                      |
|  1. 读取/接收开发计划                                                 |
|  2. 提取任务列表并排序（依赖关系）                                     |
|  3. 逐个 dispatch 子代理执行                                          |
|  4. 处理子代理返回状态                                                |
|  5. 执行两阶段审查                                                    |
|  6. 标记完成，继续下一个任务                                           |
|                                                                      |
|  +----------+   +----------+   +----------+                         |
|  | Task #1  |-->| Task #2  |-->| Task #3  |--> ...                  |
|  | 子代理A   |   | 子代理B   |   | 子代理C   |                         |
|  +----+-----+   +----+-----+   +----+-----+                         |
|       |              |              |                                |
|       v              v              v                                |
|  +---------+   +---------+   +---------+                             |
|  |合规审查  |   |合规审查  |   |合规审查  |  <-- Phase 1: 引擎规则     |
|  |质量审查  |   |质量审查  |   |质量审查  |  <-- Phase 2: 代码质量     |
|  +---------+   +---------+   +---------+                             |
+---------------------------------------------------------------------+
```

## 触发条件

**同时满足以下条件时触发**：

1. **多任务信号**：用户需求包含 3 个以上可独立实现的功能模块
2. **意图关键词**（满足其一）：
   - 用户说"子代理"、"subagent"、"dispatch"、"任务编排"
   - 用户说"逐模块开发"、"分步实现"、"每个模块都要审查"
   - 用户提供了一份开发计划/需求列表（3+ 项）

**不触发**（转交其他 Skill）：

| 场景 | 转交 |
|------|------|
| 单个功能开发（即使复杂） | `structured-dev-session` |
| 从零开始需要规划 | `game-dev-planner`（规划完成后再转本 Skill） |
| 代码审查（无任务编排） | `dev-tools-pack` → Code Review |
| 自动化重复模式 | `auto-workflow` |
| 简单修改、配置更新 | 直接修改，无需本 Skill |

## 流程详解

### Phase 0：计划获取

从以下来源获取开发计划：
- 用户直接提供的功能列表
- `game-dev-planner` 生成的 GDD 中的开发节点
- 用户描述的多功能需求（AI 自动拆解）

**计划格式**（内部标准化）：

```
任务列表：
+-- Task 1: [功能名] -- [简述] -- 依赖: 无
+-- Task 2: [功能名] -- [简述] -- 依赖: Task 1
+-- Task 3: [功能名] -- [简述] -- 依赖: 无
+-- Task 4: [功能名] -- [简述] -- 依赖: Task 2, Task 3
```

**依赖排序规则**：
- 无依赖的任务优先执行
- 有依赖的任务等待前置任务完成
- 互不依赖的任务**按顺序串行执行**（UrhoX 沙箱不支持并发文件写入）

### Phase 1：任务 Dispatch（子代理执行）

使用 **Task 工具** dispatch 子代理执行每个任务。

**Dispatch 模板**：

```
你是一个 UrhoX Lua 游戏开发专家。请完成以下开发任务：

## 任务
{task_description}

## 项目上下文
- 项目结构：{project_structure}
- 相关已有文件：{related_files}（已读取内容附在下方）
- 引擎版本：UrhoX（Lua 5.4）

## 已有代码（相关模块）
{existing_code_content}

## 技术约束
- 代码写入 scripts/ 目录
- 遵循 UrhoX 引擎规则（详见下方）
- 使用标准模块模式：local M = {} / return M
- nil 初始化变量必须加 ---@type 类型标注
- 数组索引从 1 开始
- 事件数据用 eventData["Key"]:GetType() 或 eventData:GetType("Key")

## 引擎关键规则
1. 长度单位是米，Y-up 左手坐标系
2. 程序化材质用 PBRNoTexture 系列
3. NanoVG 必须在 NanoVGRender 事件中渲染
4. UI 使用 urhox-libs/UI（原生 UI 已废弃）
5. 枚举值用常量名不用数字
6. 文件路径从资源根开始（不加 assets/ 或 scripts/ 前缀）

## 输出要求
1. 写入完整的 Lua 文件到 scripts/ 目录
2. 确保代码可直接运行
3. 提供简要说明：做了什么、改了哪些文件、需要注意什么
```

**子代理类型选择**：

| 任务复杂度 | 子代理类型 | 说明 |
|-----------|-----------|------|
| 机械实现（数据类、配置文件、简单模块） | `haiku` | 快速、低成本 |
| 标准功能（游戏系统、UI 面板、交互逻辑） | `sonnet` | 平衡质量和速度 |
| 架构设计（核心系统、模块集成、复杂逻辑） | `opus` | 最高质量 |

### Phase 2：子代理状态处理

子代理执行完毕后返回结果，按以下状态分类处理：

| 状态 | 含义 | 处理方式 |
|------|------|---------|
| **DONE** | 任务完成 | 进入两阶段审查 |
| **DONE_WITH_CONCERNS** | 完成但有疑虑 | 读取疑虑点，评估是否需要修改，进入审查 |
| **NEEDS_CONTEXT** | 缺少上下文信息 | 补充上下文（读取相关文件），重新 dispatch |
| **BLOCKED** | 任务被阻塞 | 评估原因，手动干预或跳过 |

**状态判断规则**：
- 子代理明确说"完成"且无遗留问题 → `DONE`
- 子代理说"完成但建议..."或"有一个潜在问题" → `DONE_WITH_CONCERNS`
- 子代理说"需要知道..."或"找不到..." → `NEEDS_CONTEXT`
- 子代理报错或说"无法完成" → `BLOCKED`

### Phase 3：两阶段审查

**审查顺序严格固定：先合规，再质量。合规不通过不得进入质量审查。**

#### 阶段 A：引擎规则合规审查

检查子代理产出的代码是否符合 UrhoX 引擎规则：

```
## 合规审查清单

### P0 — 致命问题（必须修复）
- [ ] 代码是否写入 scripts/ 目录？
- [ ] 是否使用了正确的 require 路径？
- [ ] NanoVG 渲染是否在 NanoVGRender 事件中？
- [ ] UI 是否使用 urhox-libs/UI（非原生 UI）？
- [ ] nil 变量是否添加了 ---@type 标注？
- [ ] 数组索引是否从 1 开始？
- [ ] 枚举是否使用常量名而非数字？
- [ ] 事件数据是否用正确的访问方式？

### P1 — 重要问题（应该修复）
- [ ] 资源路径是否从资源根开始（无多余前缀）？
- [ ] 模块是否使用 local M = {} / return M 模式？
- [ ] 是否有 Cleanup 函数释放资源和取消事件？
- [ ] 坐标系是否正确（Y-up 左手）？
- [ ] 材质 Technique 是否使用 PBRNoTexture 系列？

### P2 — 建议优化
- [ ] 配置是否集中在 CONFIG 表中？
- [ ] 是否有适当的日志输出（首次交付）？
```

**合规审查失败处理**：
1. 列出所有不合规项
2. 重新 dispatch 子代理修复（附带具体修复指令）
3. 再次审查，直到 P0/P1 全部通过

#### 阶段 B：代码质量审查

合规通过后，检查代码质量：

```
## 质量审查清单

### 可读性
- [ ] 变量和函数命名是否有意义（非 a, b, temp）？
- [ ] 复杂逻辑是否有注释说明？
- [ ] 函数长度是否合理（< 50 行）？

### 可维护性
- [ ] 是否存在重复代码（应抽取公共函数）？
- [ ] 模块职责是否单一？
- [ ] 依赖关系是否清晰（无循环依赖）？

### 健壮性
- [ ] 关键参数是否有 nil 检查？
- [ ] 除法运算是否有除零保护？
- [ ] 表访问是否有安全检查？

### 性能
- [ ] Update 循环中是否有不必要的对象创建？
- [ ] 是否有应该缓存的重复计算？
- [ ] 字符串拼接是否使用 table.concat？
```

**质量审查失败处理**：
1. 列出问题和改进建议
2. 评估严重程度：
   - 严重问题 → 重新 dispatch 子代理修复
   - 轻微问题 → 记录但不阻塞，继续下一个任务

### Phase 4：任务完成与状态更新

```
任务 #N 完成审查
  |
  v
更新 TodoWrite 标记完成
  |
  v
检查是否有依赖此任务的后续任务
  |
  +-- 有 --> 解锁后续任务，继续 Phase 1
  +-- 无 --> 检查是否还有未完成任务
              +-- 有 --> 继续下一个无依赖任务
              +-- 无 --> 进入最终集成构建
```

## 最终集成

所有任务完成后执行：

1. **构建验证**：调用 UrhoX 构建工具构建整个项目
2. **集成检查**：确认模块间引用正确、无遗漏依赖
3. **交付报告**：汇总所有任务的完成情况和审查结果

```
## 交付报告

### 任务完成情况
| # | 任务 | 状态 | 审查结果 | 备注 |
|---|------|------|---------|------|
| 1 | XXX | 完成 | 合规 OK 质量 OK | - |
| 2 | YYY | 完成 | 合规 OK 质量 Warning | 有轻微问题已记录 |
| 3 | ZZZ | 完成 | 合规 OK 质量 OK | - |

### 构建结果
- 构建状态：成功
- 文件列表：scripts/Systems/XXX.lua, scripts/Systems/YYY.lua, ...

### 遗留问题
- （列出质量审查中记录的轻微问题）
```

---

## Red Flags — 永远不要

| # | 反模式 | 为什么危险 |
|---|--------|-----------|
| 1 | 跳过合规审查直接做质量审查 | 合规问题会导致运行时崩溃 |
| 2 | 有未修复的 P0 问题时继续下一个任务 | 后续任务可能依赖有问题的代码 |
| 3 | 让子代理自己读整个项目 | 子代理上下文有限，应由主代理提供精确上下文 |
| 4 | 同时 dispatch 多个写入同一文件的子代理 | 沙箱下文件写入冲突 |
| 5 | 忽略子代理的 NEEDS_CONTEXT 状态 | 缺少上下文的代码质量无法保证 |
| 6 | 接受"差不多合规"的代码 | 引擎规则没有灰色地带，不合规就是不合规 |
| 7 | 子代理完成后不构建验证 | 语法错误和资源引用问题只有构建才能发现 |

## 使用示例

### 场景：用户提供了一份游戏功能列表

```
用户："帮我实现这些功能：
1. 背包系统 - 物品管理和 UI
2. 商店系统 - 购买/出售
3. 存档系统 - 保存/加载游戏进度
4. 敌人 AI - 巡逻/追击/攻击
5. 任务系统 - 接取/完成/奖励"
```

**编排器执行流程**：

```
1. 解析任务列表 -> 5 个独立任务
2. 分析依赖关系：
   - Task 1 (背包): 无依赖
   - Task 2 (商店): 依赖 Task 1（需要背包 API）
   - Task 3 (存档): 无依赖
   - Task 4 (敌人 AI): 无依赖
   - Task 5 (任务系统): 依赖 Task 1（奖励写入背包）

3. 执行顺序：Task 1 -> Task 3 -> Task 4 -> Task 2 -> Task 5
   （无依赖任务优先，有依赖任务等前置完成）

4. 逐个 dispatch：
   Task 1: dispatch(haiku)  -> 合规审查 OK -> 质量审查 OK -> 完成
   Task 3: dispatch(haiku)  -> 合规审查 OK -> 质量审查 OK -> 完成
   Task 4: dispatch(sonnet) -> 合规审查 OK -> 质量审查 Warn -> 记录 -> 完成
   Task 2: dispatch(sonnet) -> 合规审查 OK -> 质量审查 OK -> 完成
   Task 5: dispatch(sonnet) -> 合规审查 OK -> 质量审查 OK -> 完成

5. 最终集成构建 -> 交付报告
```

## 子代理 Dispatch 速查

详细的 dispatch 模板和上下文准备指南见 → `references/dispatch-templates.md`

### 快速 Dispatch 模式

```
Task 工具调用参数：
- subagent_type: "general-purpose" (默认) 或 "Bash" (纯命令行任务)
- model: "haiku" / "sonnet" / "opus" (根据任务复杂度选择)
- prompt: 使用标准 dispatch 模板（见 Phase 1）
- description: "Task N: {功能名}"
```

### 上下文准备原则

1. **不要让子代理自己搜索文件** — 主代理提前读取相关文件并嵌入 prompt
2. **提供精确的项目结构** — 只列出相关目录和文件，不列全部
3. **附带已完成任务的 API** — 如果后续任务依赖前置任务，附带前置模块的公共接口
