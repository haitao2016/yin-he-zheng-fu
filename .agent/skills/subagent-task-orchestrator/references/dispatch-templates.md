# Dispatch Templates — 子代理调度模板

> 本文档提供子代理 dispatch 的标准化模板，确保每次调度的上下文完整、约束明确。

---

## 模板 1：标准功能模块实现

适用于独立的游戏子系统模块（背包、商店、存档等）。

```
你是一个 UrhoX Lua 游戏开发专家。请完成以下开发任务：

## 任务
实现 {模块名称} 模块。

功能需求：
{功能描述，逐条列出}

## 项目上下文
当前项目结构：
```
scripts/
  main.lua           -- 入口文件
  {已有文件列表}
```

引擎版本：UrhoX（Lua 5.4）

## 已有代码（相关模块接口）
{粘贴前置依赖模块的公共接口，如：}
-- scripts/Systems/Inventory.lua 公共接口：
-- M.Init(scene, opts)     初始化背包
-- M.AddItem(itemId, count) 添加物品
-- M.RemoveItem(itemId, count) 移除物品
-- M.GetItems()             获取所有物品
-- M.Cleanup()              清理

## 技术约束
1. 代码写入 scripts/ 目录
2. 模块模式：local M = {} / return M
3. nil 初始化变量必须加 ---@type 类型标注
4. 数组索引从 1 开始
5. 事件数据用 eventData["Key"]:GetType() 或 eventData:GetType("Key")
6. 枚举使用常量名（如 MOUSEB_LEFT），不用数字
7. 资源路径从资源根开始，不加 assets/ 或 scripts/ 前缀
8. UI 使用 urhox-libs/UI 组件（原生 UI 已废弃）
9. 程序化材质使用 PBRNoTexture 系列 Technique
10. NanoVG 渲染必须在 NanoVGRender 事件中

## 输出要求
1. 写入完整的 Lua 文件到 scripts/ 对应目录
2. 使用 local M = {} / return M 模块模式
3. 提供 M.Init() 和 M.Cleanup() 函数
4. 配置集中在文件顶部的 CONFIG 表中
5. 添加适当的日志输出（print 语句）
6. 最后说明：做了什么、改了哪些文件、需要注意什么
```

## 模板 2：UI 面板实现

适用于 HUD、菜单、弹窗等 UI 模块。

```
你是一个 UrhoX Lua 游戏开发专家。请完成以下 UI 开发任务：

## 任务
实现 {面板名称} UI 面板。

界面需求：
{UI 描述，包含布局、交互、数据绑定}

## UI 系统说明
使用 urhox-libs/UI 组件库（Yoga Flexbox 布局 + NanoVG 渲染）：
- UI.Init() 在 main.lua 中已调用（字体和缩放已配置）
- 使用 UI.Panel, UI.Label, UI.Button, UI.Image 等组件
- 布局属性：width, height, flexDirection, justifyContent, alignItems
- 尺寸单位：数字为像素，字符串百分比如 "100%"

## 项目上下文
{项目结构和已有代码}

## 技术约束
1. 代码写入 scripts/ 目录
2. UI 必须使用 urhox-libs/UI 组件，不要使用原生 UI
3. 如果需要 UI.Init()，确认 main.lua 中已初始化
4. nil 初始化变量必须加 ---@type 类型标注
5. 提供面板的 Show/Hide/Toggle 函数

## 输出要求
1. 写入 UI 面板文件到 scripts/UI/ 目录
2. 模块模式 local M = {} / return M
3. 提供 M.Show(), M.Hide(), M.Toggle() 函数
4. 配置（颜色、字号、间距）集中在 CONFIG 表
```

{列出每个相关模块的公共 API}

{粘贴当前 main.lua 完整内容}

## 上下文准备指南

### 准备原则

1. **主代理负责读取文件** — 在 dispatch 前用 Read 工具读取所有相关文件
2. **只提供相关文件** — 不要把整个项目塞给子代理
3. **接口优先于实现** — 依赖模块只需提供公共接口说明，不需完整代码
4. **约束必须明确** — 每个 dispatch 都带上完整的技术约束清单

### 上下文收集清单

| 准备步骤 | 说明 |
|---------|------|
| 读取项目结构 | `ls scripts/` 获取当前文件列表 |
| 读取 main.lua | 了解入口文件和初始化流程 |
| 读取依赖模块 | 前置任务产出的模块代码/接口 |
| 检查资源文件 | `ls assets/` 确认可用资源 |
| 读取配置文件 | `.project/settings.json` 检查多人/单机模式 |

### 子代理模型选择指南

| 任务特征 | 推荐模型 | 理由 |
|---------|---------|------|
| 数据类模块（CONFIG、常量表） | `haiku` | 结构简单，无需复杂推理 |
| 标准 CRUD 模块（背包、商店） | `haiku` 或 `sonnet` | 模式成熟，模板化程度高 |
| UI 面板（布局、交互） | `sonnet` | 需要理解 Flexbox 布局 |
| 游戏逻辑（AI、物理交互） | `sonnet` | 需要理解游戏设计意图 |
| 核心系统集成（多模块协调） | `opus` | 需要全局视角和架构能力 |
| 修复任务（审查不通过） | 与原任务相同 | 理解原有上下文 |
