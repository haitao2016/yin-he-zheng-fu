# GDD 模板与发布文案指南

按需阅读：当用户需要生成游戏设计文档（GDD）、TapTap 商店文案、开发日志或推广文案时参考。

---

## 一、游戏设计文档（GDD）模板

### 完整 GDD 结构

```markdown
# [游戏名称] — 游戏设计文档

## 1. 概述
- **一句话描述**: [一句话概括核心体验]
- **游戏类型**: [平台跳跃 / 射击 / RPG / 益智 / 卡牌 / ...]
- **屏幕方向**: [横屏 / 竖屏]
- **目标平台**: [移动端 / PC / 双端]
- **核心体验关键词**: [3-5 个词，如"爽快、策略、收集"]

## 2. 核心玩法
### 游戏循环
[输入] → [核心动作] → [反馈] → [目标推进]

### 操作方式
| 输入 | 动作 |
|------|------|
| 触摸/点击 | ... |
| 滑动 | ... |
| 长按 | ... |

### 胜负条件
- **胜利**: ...
- **失败**: ...

## 3. 游戏系统
### 系统列表
| 系统 | 职责 | 与其他系统的关系 |
|------|------|----------------|
| 战斗系统 | ... | 读取角色数据，输出伤害 |
| 背包系统 | ... | 提供装备给战斗系统 |
| ... | ... | ... |

### 系统交互图（文字描述）
[系统 A] --数据流--> [系统 B] --触发--> [系统 C]

## 4. 进度与经济
### 成长路径
[新手引导] → [核心循环] → [中期目标] → [后期内容]

### 资源体系
| 资源 | 获取方式 | 消耗方式 |
|------|---------|---------|
| 金币 | 关卡掉落 | 商店购买 |
| 经验 | 战斗获取 | 角色升级 |

## 5. 内容规划
### 关卡/场景
| 编号 | 名称 | 核心机制 | 难度 |
|------|------|---------|------|
| 1-1 | 新手村 | 基础操作教学 | 简单 |
| 1-2 | ... | ... | ... |

### 角色/物品
| 名称 | 类型 | 属性 | 获取方式 |
|------|------|------|---------|

## 6. 技术方案
### 脚手架选择
- [ ] scaffold-2d（2D 休闲）
- [ ] scaffold-2d-physics（2D 物理平台跳跃）
- [ ] scaffold-3d-scene（3D 场景展示）
- [ ] scaffold-3d-character（3D 角色游戏）

### 模块划分
```
scripts/
  main.lua          -- 入口
  Systems/           -- 子系统
  UI/                -- 界面
  Config/            -- 配置数据
```

### 关键技术点
| 技术需求 | 解决方案 |
|---------|---------|
| 物理碰撞 | Box2D / 3D Physics |
| UI 系统 | urhox-libs/UI |
| 数据持久化 | File API / clientCloud |
| 动画 | FSM 状态机 / NanoVG |

## 7. 里程碑
### MVP（最小可玩版本）
- [ ] 核心循环可玩
- [ ] 基础 UI（菜单、HUD）
- [ ] 至少 1 个完整关卡

### 迭代 1
- [ ] 功能 A
- [ ] 功能 B

### 迭代 2
- [ ] 功能 C
- [ ] 功能 D
```

### 信息收集清单

在生成 GDD 前，必须确认以下信息（未提供则主动询问）：

| 信息 | 必要性 | 默认值 |
|------|--------|--------|
| 游戏类型 | 必须 | - |
| 核心玩法一句话 | 必须 | - |
| 屏幕方向 | 必须 | 横屏 |
| 单机/多人 | 必须 | 单机 |
| 目标平台 | 可选 | 移动端+PC |

**输出位置**: `docs/gdd.md`

## 二、TapTap 商店文案模板

### 描述结构

```
[一句话 Hook — 吸引注意力，制造好奇心]

【核心玩法】
简述核心玩法循环，2-3 句话。使用玩家视角，描述体验而非技术。

【游戏特色】
★ 特色1 — 一句话描述（聚焦体验）
★ 特色2 — 一句话描述（差异化卖点）
★ 特色3 — 一句话描述（情感共鸣）

【适合谁玩】
目标玩家群体描述，1-2 句话。
```

### 写作原则

| 原则 | 好例子 | 差例子 |
|------|--------|--------|
| 玩家视角 | "掌控每一次跳跃的节奏" | "实现了精确的物理跳跃系统" |
| 具体生动 | "一刀斩落三个水果" | "具有高伤害的攻击" |
| 简洁有力 | "冲刺、跳跃、斩杀" | "本游戏具有冲刺和跳跃功能" |
| 动词优先 | "探索、收集、建造" | "丰富的探索和收集系统" |

### taptap_publish JSON 配置

```json
{
  "taptap_publish": {
    "title": "游戏名称（简短有力，2-6 字最佳）",
    "description": "商店描述文案",
    "category": "分类",
    "screen_orientation": "landscape 或 portrait",
    "trial_note": "开发者备注（可选）"
  }
}
```

**支持的分类**:
rpg, casual, action, strategy, simulation, trivia, arcade,
adventure, card, sports, racing, puzzle, educational, music, word, board

### 分类选择建议

| 游戏特征 | 推荐分类 |
|---------|---------|
| 回合制战斗+剧情 | rpg |
| 快节奏动作 | action |
| 消消乐/匹配 | puzzle |
| 经营/建造 | simulation |
| 休闲小游戏 | casual |
| 竞速 | racing |
| 卡牌收集+策略 | card |
| 射击 | action |
| 平台跳跃 | arcade |
| 文字冒险 | adventure |

## 三、开发日志模板

### 分类标签

| 标签 | 含义 | 适用场景 |
|------|------|---------|
| `[新增]` | 新功能 | 添加了新的游戏系统或特性 |
| `[修复]` | Bug 修复 | 解决了已知问题 |
| `[优化]` | 性能/体验优化 | 提升了运行效率或用户体验 |
| `[调整]` | 数值/配置变更 | 修改了游戏平衡性参数 |
| `[重构]` | 代码结构调整 | 不改变功能的结构性改动 |
| `[美术]` | 视觉相关更新 | 贴图、模型、UI 美化 |

### 日志格式

```markdown
## 开发日志 — YYYY-MM-DD

### 本次更新摘要
一句话总结核心变更。

### 变更详情
- [新增] 功能描述
- [修复] 修复内容描述
- [优化] 优化内容描述

### 技术备注
技术决策或架构变更说明（如有）。

---
```

### 生成流程

1. 阅读 `scripts/` 下最近修改的文件
2. 对比前后变更（如有 git 记录）
3. 按标签分类整理
4. 生成结构化日志追加到 `docs/devlog.md`

**输出位置**: `docs/devlog.md`（追加到文件末尾）

## 四、推广文案模板

### 文案类型模板

#### 游戏推介（系列文案）

```
【推文 1/3 — 核心玩法】
[动词开头，描述核心动作]。[一句话说明玩法循环]。
[设问或反转，制造好奇心]

【推文 2/3 — 差异化卖点】
[与同类游戏的不同之处]。[具体的游戏场景描述]。
[情感共鸣句]

【推文 3/3 — 号召行动】
[总结核心体验]。[制造紧迫感或期待感]。
[号召语：现在就来体验 / 等你挑战]
```

#### 版本更新公告

```
## [版本号] 更新公告

### 新增内容
- 新功能 1：一句话描述
- 新功能 2：一句话描述

### 优化改进
- 优化点 1
- 优化点 2

### 已修复
- 修复了 XXX 的问题

---
快来体验新版本！[号召语]
```

#### 特色功能介绍

```
【功能名称】
一句话概括功能核心价值。

在 [游戏场景] 中，你可以 [具体操作]。
[操作的结果和反馈]，带来 [情感体验]。

[一句话总结，呼应标题]
```

### 写作原则

1. **玩家视角**: 描述体验而非技术实现
2. **具体生动**: 用场景化描述替代抽象概念
3. **简洁有力**: 每句不超过 20 字
4. **动词优先**: "冲刺、跳跃、斩杀" 优于 "拥有冲刺功能"
5. **无 emoji**: 除非用户明确要求
6. **不承诺时间**: 不写"即将上线"、"下周更新"等时间承诺
